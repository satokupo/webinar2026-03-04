# planmode-rules（プランモードルール）

## 概要

Claude Code のプランニングワークフロー全体を管理するスキル。
方針を「別のセッションでもそのまま実行できるプランファイル」に落とし込み、複数エージェントで検証するところまでを受け持つ。

このスキル単体でもブレスト（Phase 1）を内包しているため、要件が比較的はっきりしている場合やそこまで深い議論が不要な場合は、最初からこのスキルだけで回すことができる。
一方、多角的な視点でしっかり議論してから方針を固めたい場合は、summon-experts でパネルディスカッションを先に行い、その結果を引き継いで Phase 2 以降に合流させるのが推奨ルート。

直接呼び出しはできない（`user-invocable: false`）。
`EnterPlanMode` ツールが実行されると PostToolUse フックが発動し、「planmode-rules を読み込め」という指示が自動注入される仕組み。

スキルの責任範囲は「プランの作成・検証まで」と明示されており、プラン実行後のセッションには影響を及ぼさない。

---

## 内部フロー

### Phase 1: ブレスト・検討

ユーザーと対話しながら要件を明確化するフェーズ。この段階ではプランファイルに触れない。

summon-experts を先に使った場合はこのフェーズがディスカッション結果で置き換わるため、実質スキップまたは軽い補足確認だけで Phase 2 に進む。
summon-experts を使わずにこのスキルから始めた場合は、ここで通常の対話ベースのブレストを行う。

```
ルート A（summon-experts 経由）:
  入力: ディスカッション結果 + 最終提案
  → 補足確認のみで Phase 2 へ

ルート B（このスキルから直接開始）:
  入力: ユーザーの要件・やりたいこと
  → 対話で要件を掘り下げ
  → 「プラン作成フェーズに入っていいですか？」と確認して Phase 2 へ
```

---

### Phase 2: 合意形成（YAML構造化）

ブレスト結果を構造化し、ユーザーとの認識ズレをなくすフェーズ。**必須プロセス**として設計されており、AI 側の判断で省略することはできない。

```
入力: Phase 1 の対話結果
  ↓
処理:
  1. ユーザーの要望を YAML 構造化して提示（キーは日本語）
  2. AI の採用アプローチも YAML 構造化
  3. ユーザーの OK を待つ
  ↓
出力: AskUserQuestion で次の選択肢を提示
  選択肢:
    1. コンテキストを圧縮してからプラン作成（推奨）→ Phase 3 Step 0 を経由
    2. 直接プラン作成 → Phase 3 Step 0 をスキップ
```

---

### Phase 3: プランファイル作成

**Step 0: コンテキスト引き継ぎ（Phase 2 で「圧縮する」を選んだ場合のみ）**

Phase 1〜2 のブレスト会話は情報密度が低い。これを高密度な引き継ぎファイルに凝縮してから `/compact` を送信し、コンテキストをリフレッシュする。

```
処理:
  1. context-handover スキルを発動 → 引き継ぎファイル生成
  2. /compact 自動送信 → コンテキスト圧縮
  3. SessionStart フックが引き継ぎファイルを先頭に自動注入
  ↓
→ 圧縮後、planning-flow.md を再度読み込んで続行
```

**関連スキル**: [context-handover](../02_サブスキル/context-handover.md)

**プランファイルの書き出し手順:**

```
処理:
  1. EnterPlanMode を呼び出し、システムにパスを提示させる
  2. パス検証（.claude/plans/ 配下であることを確認）
  3. テンプレート（plan-skeleton.md）を cp でコピー
  4. current-session-id スキルでセッション ID を取得
  5. Edit でプレースホルダーを置換してプラン内容を書き込む
  6. ExitPlanMode
```

**関連スキル**: [current-session-id](../02_サブスキル/current-session-id.md)

**プランファイルの固定構造:**

```markdown
---
planning-session: {セッションID}
created: {日時}
---
# タイトル

## 📝 背景
## 📄 ファイル操作
## 🔧 変更内容
## ✅ 検証方法（省略可）
## 📌 スコープ外の課題（省略可）
## ✅ 実装完了チェック（定型文）  ← impl-checker エージェント起動の指示
## 🚀 コミット（定型文）          ← git-commit-rules スキル起動の指示
```

フッターの2セクションは定型文で、プラン実行者（未来の別セッション）が実装完了後にそのまま従う手順書になっている。

**関連エージェント**: [impl-checker](../02_サブスキル/impl-checker.md)
**関連スキル**: [git-commit-rules](../02_サブスキル/git-commit-rules.md)

---

### Phase 4: プラン検証

作成したプランを複数エージェントの自動再帰ループで検証する。最も複雑なフェーズ。

**発動キーワード:**

| キーワード | 動作 |
|-----------|------|
| 「検証準備」 | Phase 4 を開始（デフォルトで Opus 検証） |
| 「Sonnetで検証準備」 | Opus 側エージェントを Sonnet に切り替えて開始 |
| 「完璧ですか？」「これで完璧？」 | 中断後の検証ループ再開 |

「検証準備」という文字列をキーワードとして埋め込むことで、ユーザーの発話意図を AI が確実に認識し、前処理→エージェント起動→自動ループという一連のフローに正しく遷移する。

```
Step 1: 前処理
  - extract_conversation.py でセッション JSONL からテキスト発言のみを抽出
  - 出力: {session_id}_conversation.md（.claude/temp/ に保存）

Step 2: モデル割り当て決定
  - デフォルト: Opus
  - 「Sonnetで検証準備」と言えば Opus 側を Sonnet に切替可能

Step 3: 検証エージェント 3並列起動（1レスポンス内で同時発行）

  ┌─ Opus側エージェント ─────────────────────────────────┐
  │ 入力: conversation.md + プランファイル              │
  │ 担当: 観点1（会話ニュアンス）                       │
  │       観点2（新規セッション実行可能性）             │
  │       観点5（サブエージェント設計）                 │
  │       + 条件付き観点（該当時のみ）                 │
  └────────────────────────────────────────────────────┘
  ┌─ Sonnet側エージェント ──────────────────────────────┐
  │ 入力: プランファイルのみ                            │
  │ 担当: 観点3（指示の具体性）                         │
  │       観点4（影響範囲の網羅性）                     │
  └────────────────────────────────────────────────────┘
  ┌─ Haiku側エージェント ───────────────────────────────┐
  │ 入力: plan-skeleton.md + プランファイル             │
  │ 担当: 観点6（テンプレートフォーマット準拠）         │
  │ ※ ステートレスチェックのため毎ラウンド新規起動     │
  │   （resume なし・エージェントID保持なし）          │
  └────────────────────────────────────────────────────┘

Step 4: メイン Opus が全エージェントの報告を統合・評価
  - 採用/却下を判断
  - 横断チェック（各エージェントの報告に矛盾がないか確認）

Step 5: 修正が必要なエージェントにのみ resume で対話
  - 両方要修正 → 並列 resume
  - 片方のみ → 該当エージェントのみ resume
  - 全OK → Step 6 へ
  ※ Haiku側は resume 対象外。観点6が非準拠の場合は
    メインが直接プランを修正し、次ラウンドで新規 Haiku を起動して再確認

Step 6: 収束判定
  - 収束条件: 全観点（1〜6）OK + 反論なし
             ※ Haiku の観点6（フォーマット準拠）も必須
  - 中断条件: 同一争点の繰り返し（2回却下後も再主張）or ラウンド上限5回
  - 中断時 → ユーザーにエスカレーション

Step 7: 最終警戒チェック（スキップ禁止）
  - 既存の検証エージェントとは別に、新規 Opus サブエージェントを起動
  - 入力: プランファイル + 変更対象ファイル一覧のみ
  - 渡さないもの: conversation.md、検証ループの結果（先入観を排除）
  - チェック: ファイル存在性・内容整合性・Git追跡状態・参照整合性

Step 8: 最終報告（2部構成テーブル）
  Part 1: 検証エージェント結果（観点1〜6）
  | 観点 | 担当 | 結果 |
  |------|------|------|
  | 1. 会話ニュアンス | Opus | ✅ OK |
  | 2. 実行可能性 | Opus | ✅ OK |
  | 3. 指示の具体性 | Sonnet | ✅ OK |
  | 4. 影響範囲の網羅性 | Sonnet | ✅ OK |
  | 5. サブエージェント設計 | Opus | ✅ OK |
  | 6. フォーマット準拠 | Haiku | ✅ OK |

  Part 2: 最終警戒チェック結果（Step 7 の新規 Opus による）

  - 収束時: 「🔥🔥 完璧です 🔥🔥」+ サウンド再生
  - Phase 5（Codex レビュー）の案内を固定テンプレートで出力
```

---

### Phase 5: Codex レビュー（オプション）

自動遷移・自動推奨は禁止。ユーザーが明示的に指示した場合のみ起動する。

**発動キーワード:**

| キーワード | 動作 |
|-----------|------|
| 「CODEX検証」 | Phase 5 を開始 |
| 「Codexにレビューして」「Codexレビューお願い」 | 同上（自然言語での指示も可） |

Phase 4 完了（「🔥🔥 完璧です 🔥🔥」判定済み）が前提条件。

```
Step 1: codex_reviewer.py で Codex CLI を live モード起動
Step 2: 各指摘を「採用 / 却下 / 確認」でトリアージし、resume で Codex と対話
Step 3: 追加質問（最大2回）
Step 4: 修正規模に応じて3段階分岐（軽微/中程度/大幅）して報告
  ↓
完了時: 「🛡️🛡️ Codex によるレビューが完了しました 🛡️🛡️」+ サウンド再生
```

## フロー全体図

```
Phase 1       Phase 2         Phase 3                 Phase 4              Phase 5
ブレスト  →   合意形成YAML →  プランファイル作成   →   プラン検証        →  Codexレビュー
 ↕対話        ↕ユーザー承認    context-handover         3並列エージェント     （オプション）
              YAML構造化       current-session-id       ↕自動再帰ループ
                               plan-skeleton.md         最終警戒チェック
                               impl-checker定型文       play_sound.py
                               git-commit-rules定型文
```

## このスキルで工夫したポイント

<div style="padding: 0 56px; display: flex; flex-direction: column; gap: 16px;">

<div style="border: 1px solid #e2e8f0; border-radius: 8px; overflow: hidden;">
<h3 style="margin: 0; padding: 12px 20px; font-size: 16px; font-weight: 700; border-bottom: 1px solid #bfdbfe; background: #dbeafe; color: #1d4ed8;">プラン作成前の明示的なコンパクション（Phase 3 Step 0）</h3>
<div style="padding: 12px 20px 16px;">

ブレスト会話は試行錯誤を含むため情報密度が低い。そのままプラン作成に入ると、重要な決定事項がコンテキストの中間に埋もれる（lost-in-the-middle）だけでなく、その後の検証フェーズでコンテキストが溢れるリスクもある。

Phase 3 に入る前に context-handover で会話を高密度な引き継ぎファイルに凝縮し、明示的にコンパクションを1回挟む。
これにより、プラン作成から検証完了までコンテキストが溢れにくくなり、検証の多段ループも安定して回せる。

</div>
</div>

<div style="border: 1px solid #e2e8f0; border-radius: 8px; overflow: hidden;">
<h3 style="margin: 0; padding: 12px 20px; font-size: 16px; font-weight: 700; border-bottom: 1px solid #bfdbfe; background: #dbeafe; color: #1d4ed8;">サブエージェント用テンプレートの外部ファイル化</h3>
<div style="padding: 12px 20px 16px;">

検証エージェントに渡すプロンプトテンプレートや条件付き観点の定義は、メインのスキルプロンプトには埋め込まず、`references/` や `conditions/` ディレクトリに分離してパス指定で読み込む構造にしている。

メリットは2つ。メインのコンテキストがテンプレート文言で膨れないこと、そしてテンプレートの調整がしやすいこと。検証観点を追加・修正したくなったときに、該当ファイルだけ編集すればよく、スキル本体を触る必要がない。

</div>
</div>

<div style="border: 1px solid #e2e8f0; border-radius: 8px; overflow: hidden;">
<h3 style="margin: 0; padding: 12px 20px; font-size: 16px; font-weight: 700; border-bottom: 1px solid #bfdbfe; background: #dbeafe; color: #1d4ed8;">安全寄りの多段検証</h3>
<div style="padding: 12px 20px 16px;">

全体を通して「やりすぎ」なくらい検証を重ねる設計にしている。

- **3並列エージェント**で6つの観点を検証（Phase 4 Step 3）
- **確証バイアスの排除**: 検証ループで全観点 OK が出た後に、そのループ結果を一切知らない新規エージェントで最終チェックを行う（Phase 4 Step 7）。先入観を構造的に排除する仕組み
- **impl-checker の固定埋め込み**: プランのフッターに「実装完了後に impl-checker エージェントを起動してプラン通り実装できたか確認する」という手順を定型文として入れてしまっている。プラン実行者（未来の別セッション）が自動的にこの確認を行う構造

</div>
</div>

<div style="border: 1px solid #e2e8f0; border-radius: 8px; overflow: hidden;">
<h3 style="margin: 0; padding: 12px 20px; font-size: 16px; font-weight: 700; border-bottom: 1px solid #bfdbfe; background: #dbeafe; color: #1d4ed8;">作業内容に応じたモデルの使い分け</h3>
<div style="padding: 12px 20px 16px;">

検証の6つの観点を、モデルの得意分野に応じて割り振っている。

- **Opus 担当**: 会話ニュアンスの読み取り、実行可能性の判断、サブエージェント設計 — 判断力・推論力が必要な観点
- **Sonnet 担当**: 曖昧表現の検出、影響範囲の網羅性 — 機械的にチェックできる観点
- **Haiku 担当**: テンプレートフォーマット準拠 — ステートレスな構造比較タスク

Sonnet 側には conversation.md を渡さない（プランファイルのみ）。「プラン単体で完結しているか」を会話コンテキストなしで検証させる設計。

Haiku 側は毎ラウンド新規起動する。テンプレートとプランを読んで一致を確認するだけなので、対話的なフィードバックループのコンテキストを保持する必要がなく、resume は不要。コスト最小で観点6を毎ラウンド確実にカバーする。

</div>
</div>
