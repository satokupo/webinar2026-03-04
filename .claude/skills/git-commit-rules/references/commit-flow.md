# コミットフロー

## Phase 1: コミット対象の決定と分割

`git status` で変更ファイル一覧を確認し、`git diff` で変更内容を把握する。

### コミット対象が文脈から明確な場合

プラン実行後など、セッション中の作業でコミット対象が分かっている場合:

- **このセッションで変更・作成したファイルのみ**を `git add` の対象とする
- 他セッションの変更は `git add` しない。unstaged変更が `git add` 時に干渉する場合のみ `git stash` で退避する
- 複数の論理単位がある場合は下記のグルーピング基準に従って分割する

### コミット対象が文脈から明確でない場合

ユーザーが `/commit` のみで呼び出し、特に指示がない場合:

- `git status` / `git diff` で全変更を精査する
- 下記のグルーピング基準に従って論理的な変更単位に分割する
- 各グループにコミットメッセージを割り当てる

### グルーピング基準

- **原則**: 1コミット = 1つの論理的な変更単位（目的が同じ変更をまとめる）
- **例**: `claude/skills/` 配下のスキル改修と `shell/` 配下のエイリアス追加は別コミット。同じスキルの SKILL.md と references/ の変更は同一コミット

## Phase 2: コミットメッセージの作成

- 単一コミットの場合: メッセージを1つ作成する
- 複数コミットの場合: 各グループのメッセージをそれぞれ作成する
- メッセージ形式は SKILL.md の「コミットメッセージ形式」に従う

## Phase 3: 提示と承認

- **単一コミット**: コミットメッセージをテキスト表示する
- **複数コミット**: 全グループのコミットメッセージを一覧で表示する（グループごとの対象ファイルも含める）

コミットメッセージの提示と**同一ターンで** AskUserQuestion を呼び出し、以下の固定選択肢を提示する:

| # | label | description |
|---|-------|-------------|
| 1 | コミット＆チャット保存 (Recommended) | コミット（→リモートがあればpush）→ save-chatlogでチャットログを保存 |
| 2 | コミットのみ | コミット（→リモートがあればpush）のみ実行 |

> **注意**: コミットメッセージなしに「コミットしていいですか？」と聞くのはNG。必ずメッセージの提示と AskUserQuestion による選択肢提示を1ステップで行うこと。

- **Other（自由入力）の場合**:
  - メッセージの修正指示 → 修正を反映して再度 Phase 3 の提示と承認に戻る
  - コミット中止や方針変更 → ユーザーの入力内容に応じて判断する

## Phase 4: 実行

Phase 3 で「コミット＆チャット保存」または「コミットのみ」が選択された場合に実行する。

1. 各グループを順次 `git add <file>` → `git commit` で実行する（`git add .` は使わない）
2. 各コミット後に `git rev-parse --short HEAD` でハッシュ（短縮形）を取得し、全コミット完了後にまとめて表示する:
   - **単一コミット**: `コミット完了: \`abc1234\``
   - **複数コミット**:
     ```
     コミット完了:
     - `abc1234` feat: 認証機能の追加
     - `def5678` fix: ヘッダーのスタイル修正
     ```
3. `git remote` を実行し、リモートリポジトリの有無を判定する
   - 出力がある（リモートが設定済み）→ `git push` を実行する
   - 出力が空（リモート未設定）→ push をスキップし「リモートリポジトリが未設定のためpushをスキップしました」と表示する

**選択肢別の追加処理:**

- **コミット＆チャット保存**: 全コミット完了後に Skill ツールで `save-chatlog` を発動（1回のみ）
- **コミットのみ**: 追加処理なし
