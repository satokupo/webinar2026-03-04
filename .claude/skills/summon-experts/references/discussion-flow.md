# パネルディスカッション進行フロー

## Phase 1: 課題分析と専門家ロール提案

### やること

1. これまでの会話履歴を分析し、ユーザーの課題・論点をYAML形式で整理する
2. その課題を解決するために適した専門家（3〜5人）を提案する
3. 課題YAMLと専門家ロール提案を**1回のレスポンスで同時に**提示する

### 既存成果物のリファクタリングが含まれる場合

議論の対象に、既存の成果物（スキル、設定ファイル、ルール、プロンプト等）のリファクタリング・修正・改善が含まれる場合:

- 課題YAMLの論点に「既存内容との矛盾・不整合の確認」を必ず追加する
- この論点は Phase 3 で通常の議題として議論される（専門家が対象を精読し、矛盾を探す）
- 矛盾が見つかった場合は、既存の「矛盾・課題の検出時フロー」（Phase 3）が適用される
- 矛盾がなければ「矛盾なし」としてまとめ、次の議題へ進む

### 課題YAMLのフォーマット

```yaml
課題の概要: |
  〇〇を実現したいが、△△の方針が定まっていない
背景:
  - ユーザーはブレスト中に〜という話をしていた
  - 〜という制約がある
論点:
  - 論点A: 〜をどうするか
  - 論点B: 〜と〜のどちらを選ぶか
  - 論点C: 〜のリスクをどう評価するか
ゴール: |
  各論点について専門家の多角的な議論を経て、方針を決定する
```

### 専門家ロール提案のフォーマット

課題YAMLの直後に、以下の形式で提案する：

```
## 召喚する専門家

| # | 役職名 | 召喚理由 |
|---|--------|----------|
| 1 | UXデザイナー | ユーザー体験の観点から〜を評価するため |
| 2 | バックエンドエンジニア | 実装上の制約や技術的な実現可能性を判断するため |
| 3 | プロダクトマネージャー | ビジネス要件と優先順位のバランスを取るため |
```

### ユーザーとのやりとり

- ユーザーが修正を求めた場合 → 課題YAML・専門家を修正して再提案
- ユーザーがOKした場合 → Phase 2 へ進む

---

## Phase 2: フルキャラクター設定の展開

### やること

1. Phase 1で承認された各専門家に対し、フルキャラクター設定を作成して表示する
2. **Skill ツールで `tab-rename` を発動する**（タブ名の自動設定）
   - 課題YAMLの概要から、ディスカッションテーマを表すタブ名が自動設定される（例: 課題が「認証システムの改善」→ タブ名「認証改善」）
   - リネーム完了報告は tab-rename スキルが行う。summon-experts 側では重複報告せず、テキストで確認を行う

### フォーマット

各専門家について以下を記述する：

```
### 1. UXデザイナー

- **専門領域・経歴**: モバイルアプリのUX設計を10年以上担当。大手IT企業でのデザインシステム構築経験あり
- **視点の傾向**: ユーザー体験優先。技術的制約よりも「ユーザーが迷わないか」を重視する
- **議論での立ち位置**: 推進派。良いUXのためなら工数増加も許容する立場
```

表示する項目：
- **専門領域・経歴の概要** — どんな背景を持つ専門家か
- **視点の傾向** — 何を重視するか（例: 「実装コスト重視」「ユーザー体験優先」「セキュリティ最優先」）
- **議論での立ち位置** — 議論における役割（例: 「慎重派」「推進派」「バランサー」「批判的検証者」）

### ユーザーとのやりとり

「タブ名とキャラクター設定はこれでOKですか？」とテキストで確認し、ユーザーの返答を待つ。

- ユーザーが修正を求めた場合 → 指示に従って修正し再提案（tab-rename の再発動が必要ならスキルを再発動する）
- ユーザーがOKした場合 → Phase 3 へ進む

---

## Phase 3: 議論の進行

### やること

1. **全体の議題一覧（サマリー）**を提示する
2. 「**ディスカッションを開始します。**」と宣言する
3. 議題ごとに議論を進行し、順に出力する

### 進行ルール

- **1ターンで全議題を自動進行する**。ただし議題ごとに separator.py を実行してフラッシュポイントを作り、Unicode 罫線の見出しを直接出力して区切りを作る
- 各議題の進行フロー:
  1. separator.py を Bash で実行する（フラッシュポイント — **省略厳禁**。前の議題のテキストをユーザーに表示させるための出力区切り）
  2. Unicode 罫線テンプレートで議題見出しを直接出力する（区切り＋議題開始の宣言）
  3. 議題の議論テキストを出力する（専門家の発言）
  4. 議論が終わったら、まとめをマークダウン引用ブロックで出力する
  5. 次の議題へ進み、1 に戻る
- **Unicode 罫線テンプレートが視覚的な区切りを生む**。これにより議題間が明確に分離され、ユーザーが読みやすくなる
- **基本は自動進行で会議を完走させる**。ただし「これを確認しないと議論の方向性が大きくずれる」と判断した場合に限り、議論を一時中断してAskUserQuestionで質問する。些細な確認で会議を止めない

**議題番号ルール（絶対ルール — 例外なし）**:
- 議題番号はセッション開始から単調増加する。**リセットは絶対禁止**
- ユーザーのフィードバックを受けて再議論する場合は新しい番号を付与する（「議題N 再開」ではなく「議題 N+1」）
- 関連する元議題がある場合はタイトルに括弧書きで付記する（例: 「議題7: ディレクトリ名の再検討（議題4関連）」）
- モード遷移（議論→プランモード等）を跨いでも**カウントは絶対に継続する**。振り出しに戻さない

**矛盾・課題の検出時フロー**:
- 議論中に矛盾・未解決課題を発見した場合、専門家だけで「スコープ外」と判断して先送りしてはならない
- 必ずユーザーに確認する: 「このセッションのスコープに含めますか？ それともログに残して次回以降に回しますか？」
- スコープに含める場合 → そのまま議論・対応を継続する
- ログに回す場合 → Phase 4 の推奨アクションにログファイル作成を作業項目として含める
- 以下は絶対禁止:
  - 「実務に影響がない」「別タイミングで修正」等の理由で記録なしに先送りすること
  - パネリスト間の議論の中だけで「後で対応」と流すこと

#### フラッシュポイント（separator.py）

各議題の見出しを出力する**直前**に、以下を Bash ツールで実行する:

python3 {スキルベースディレクトリ}/separator.py

- **省略厳禁**: このツール呼び出しが議題ブロック間のフラッシュポイントとなる。省略すると全議題が一括バッチ出力され、ユーザーの待ち時間と読む負荷が増大する
- separator.py の出力は空文字（折りたたまれて見えない）だが、ツール呼び出しの境界でテキストバッファがフラッシュされることが目的
- `{スキルベースディレクトリ}` は SKILL.md 読み込み時に提示される `Base directory for this skill:` のパスを使用する

#### 議題見出しの出力方法

各議題の開始時に、以下のテンプレートを Claude のテキスト出力として直接書く:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
**📋 議題N: タイトル**
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- `━`（U+2501 BOX DRAWINGS HEAVY HORIZONTAL）を使用
- 太字マークダウン（`**`）で議題タイトルを強調
- Claude のテキスト出力として直接書くため、絶対に折りたたまれない

#### まとめの出力方法

各議題の議論後、まとめをマークダウン引用ブロックで直接出力する:

> **まとめ**
> - **論点**: 各専門家の見解の要約
> - **暫定結論**: 暫定結論のテキスト
> - **事実確認**: 確認済み事項 / 未確認事項（あれば）

まとめは Claude のテキスト出力として直接書く。
引用ブロックで囲むことで議論テキストとの視覚的な区別を作る。

### 議題ごとの出力フォーマット

罫線テンプレートで見出しを出力し、次に議論、最後にまとめの引用ブロック:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
**📋 議題1: ○○について**
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**UXデザイナー**: ユーザーの動線を考えると、〜が重要だと思います。
**バックエンドエンジニア**: 実装面では〜という制約があります。ただ〜なら対応可能です。
**プロダクトマネージャー**: 両方の意見を踏まえると、〜というアプローチが現実的ですね。
**UXデザイナー**: なるほど、それなら〜という形で妥協できます。

> **まとめ**
> - **論点**: ○○について各専門家の見解が分かれた。△△派と□□派の議論の結果...
> - **暫定結論**: ◇◇する方針で進める
> - **事実確認**: ○○のAPIレート制限を公式ドキュメントで確認済み / 未確認事項なし

議論テキストの後に引用ブロックでまとめを書く。

### 議論の質に関するガイドライン

- 各専門家は**自分の専門領域の視点**から発言する。全員が同じことを言わない
- 意見の対立や建設的な議論を含める。全会一致で終わる必要はない
- 発言は具体的にする。抽象論ではなく、課題に即した具体的な提案・懸念を述べる
- 議論は**役職名のみ**で進行する（フルキャラ設定を毎回繰り返さない）
- 1つの議題につき、各専門家が少なくとも1回は発言する。ただし全員が均等に話す必要はなく、議題に関連が深い専門家が多く発言してよい

### 事実確認ルール

> 専門家ロールを演じている場合、LLMは「分からない」と言うよりも推測で「それらしい」回答を生成する傾向がある。自己検閲（発言前に止める）はLLMの逐次生成上不安定なため、他の専門家が推測を検知して指摘する相互検証方式を採用する。

#### 原則

検証可能な事実について推測で議論してはならない。推測する前にまず調べる。

#### 検知メカニズム（相互検証方式）

専門家が以下の推測表現を用いて**検証可能な事実**について発言した場合、他の専門家はそのまま受け入れてはならない。「それは確認できる事実では？ 調べましょう」と指摘し、調査を実行する。

推測表現の例:
- 「〜はず」「おそらく〜」「たぶん〜」
- 「確か〜」「〜と記憶している」「〜だと思う」（事実に対して使われた場合）
- 「一般的には〜」（具体的な根拠なし）

なお、断定的な事実主張であっても、他の専門家が正確性に疑問を感じた場合は確認を求めてよい。

#### 対象の区別

- **対象**: 検証可能な事実（ツールの仕様、APIの挙動、公式ドキュメントに記載されている設定等）
- **対象外**: 意見・設計判断（「〜がいいと思う」「〜のアプローチが適切だと思う」）
- **判断に迷った場合は調査する方に倒す**（調査してムダだったコストより、推測で議論を進めるリスクの方が大きい）

#### 調査方法

- **WebSearch で完結する軽い確認** → メインで直接実行
- **ドキュメント精読が必要な場合** → Task ツールでサブエージェント（Sonnet 推奨）に依頼し、要点を3〜5行で要約して返してもらう（コンテキスト保護）
- **調査しても確定情報が得られなかった場合** → 「調査したが未確認」としてまとめに記録し、暫定仮説として扱う（最終結論の根拠にはしない）

調査結果が得られたら、事実ベースで議論を再開する。

## Phase 4: 最終提案

### やること

全議題の議論結果を踏まえ、専門家集団としての最終提案をまとめる。

Phase 4 では **最終確認事項** と **最終提案** を1回の出力で連続して提示する。

最終提案を出力する前に、以下を確認する:
- 議論中に検出された矛盾・課題はすべてスコープ内で解消済み、またはログファイル作成が推奨アクションに含まれている
- 「別タイミングで対応」等の理由で未記録のまま先送りされた項目がない
- スコープ外とした項目がある場合、ユーザーの承認を得ている
- 推奨アクションがユーザーの元の要望を解決する因果関係が明確である

#### フラッシュポイント（separator.py）

最終確認事項・最終提案の各見出しを出力する**直前**に、以下を Bash ツールで実行する:

python3 {スキルベースディレクトリ}/separator.py

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
**📋 最終確認事項**
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase 3 の議論では、些細な確認で会議を止めない方針で自動進行する。そのため、ユーザーの明示的な承認を得ていない暫定結論や前提が残っている可能性がある。最終提案を出す前に、これらを一覧でユーザーに確認する。

#### 収集方法

Phase 3 の各議題のまとめを1つずつ振り返り、以下のカテゴリに該当する項目がないかチェックする:

- **(a)** 専門家間で意見が分かれ、暫定結論で着地した項目
- **(b)** 議論中に「ユーザーに確認すべき」と言及されたが、議論を止めずに流した項目
- **(c)** ベストプラクティスからの乖離で、ユーザーの明示的承認が未確認の項目
- **(d)** 既存ルール・設定との不整合が指摘され、対応方針が未確定の項目
- **(e)** 実装上の前提条件（パス・命名・スコープ等）で、ユーザーの意図が未確認の項目

**除外条件**: 専門家間で合意済みかつ争点がない項目は対象外。

#### 提示フォーマット

確認事項がある場合、以下の形式でテキスト出力する:

```
## 最終確認事項

議論の中で以下の確認事項が発生しました。最終提案に反映するため、確認をお願いします。

1. **〇〇について**: △△という暫定結論が出ています。（議題N）

2. **□□について**: ○○派と△△派で意見が分かれました。暫定結論は○○です。（議題M）

3. **◇◇のパス指定**: `.claude/temp/` を前提としています。（議題L）
```

- 各項目は番号付きで、**内容** + **暫定結論（あれば）** + **関連議題番号**の3要素を含める
- **項目間に空行1行を入れる**（視覚的なブロック分離のため）
- 暫定結論がある場合は必ず併記し、ユーザーが承認か変更かを判断できるようにする

#### 確認事項がゼロの場合

全議題を振り返った結果、確認事項がない場合は以下を出力する:

```
全議題の暫定結論はすべてユーザー承認済みです。確認事項はありません。最終提案に進みます。
```

この報告は省略してはならない（ゼロであっても報告義務がある）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
**📋 最終提案**
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```
## 最終提案

### 問題の要約
（ユーザーの課題を1〜2文で要約）

### 推奨アクション
1. **〇〇すること** — 理由: 〜
2. **△△すること** — 理由: 〜
3. **□□すること** — 理由: 〜

### リスク・留意点
- 〜というリスクがある。対策として〜
- 〜の点は追加検討が必要

### 議論で合意に至らなかった点
- （もしあれば）〜については意見が分かれた。〜派は〜、〜派は〜という立場

### 未解消の課題・要対応事項
- （該当がある場合）課題の内容とログファイル作成の指示
- （該当がない場合）「該当なし」と明記

### 要望充足の確認
- **ユーザーの元の要望**: （ユーザーが最初に解決したかったことを1文で要約）
- **推奨アクションによる解決**: （推奨アクションがこの要望をどう解決するかの因果関係を説明）
- **カバーできていないリスク**: （あれば記載。なければ「なし」）
```

ログファイル作成が必要な場合は、推奨アクションの項目としても記載する:
```
### 推奨アクション
N. **ログファイルを作成する** — `.claude/context/issues/YYYY-MM-DD_課題概要.md` に未解消課題を記録する
```

### スコープ外課題のログファイル仕様

未解消の課題をログに回す場合、以下のファイルを作成する。
**重要**: 別セッションの AI がこのファイルだけ読んで解決に向かえるだけの詳細さで記述すること。

- **保存先**: `<project>/.claude/context/issues/`
- **ファイル名**: `YYYY-MM-DD_課題概要.md`（例: `2026-02-19_スキル間の命名規則矛盾.md`）

フォーマット:

```
# スコープ外課題: （課題の簡潔なタイトル）

session_id: ${CLAUDE_SESSION_ID}
date: YYYY-MM-DD

## 背景・経緯
（どういう作業の中で、どういう流れでこの話が出てきたか。詳細に記述する）

## 課題の内容
（何と何が矛盾しているか、何が未解決か）

## スコープ外とした理由
（なぜ今回のセッションで対応しなかったか）

## 潜在的な影響
（放置した場合にどういう問題が起き得るか）

## 推奨する解決方針
（次のセッションで具体的に何をすべきか）

## 関連ファイル
（関係するファイルパスのリスト）
```

### 最終提案出力後のアクション提案

最終提案を出力したら、ユーザーのOK応答を待たずに、同じターンで `AskUserQuestion` ツールを呼んで次のアクションを提案する。確認事項や最終提案へのフィードバックがある場合、ユーザーは Other で入力できる。
label はシンプルな短文に留め、description で詳細を補足する設計とする。

- **question**: 「次のアクションを選択してください」
- **header**: 「次のアクション」
- **multiSelect**: false
- **options**:
  1. **label**: 「合意形成YAMLを作成する (Recommended)」
     **description**: 「planmode-rulesを発動し、議論結果をYAMLとして整理します。承認後にプラン作成の進め方を選べます」
  2. **label**: 「コンテキストを圧縮してからプラン作成」
     **description**: 「planmode-rulesを発動し、YAMLはスキップ。context-handoverで情報圧縮してからプラン作成に入ります」
  3. **label**: 「直接プラン作成」
     **description**: 「planmode-rulesを発動し、YAMLも情報圧縮もスキップして直接プラン作成に入ります」
- Other は AskUserQuestion が自動提供する（options に含めない）

#### 選択後の動作

- **選択肢1（合意形成YAML）**: `planmode-rules` スキルを Skill ツールで発動し、Phase 2（合意形成YAML）を実行する。**Phase 2 のスキップは絶対禁止。AI発の省略理由はすべて無効。** planmode-rules 発動後、planning-flow.md の Phase 2 手順に従い、YAML をユーザーに提示すること。Phase 2 末尾の AskUserQuestion でコンテキスト圧縮の要否を再度選択できる
- **選択肢2（コンテキスト圧縮してプラン作成）**: `planmode-rules` スキルを Skill ツールで発動し、「合意形成YAMLをスキップ、コンテキスト圧縮あり」と伝える。Phase 2 が省略され、Phase 3 の Step 0（context-handover）を実行してからプラン作成に入る
- **選択肢3（直接プラン作成）**: `planmode-rules` スキルを Skill ツールで発動し、「合意形成YAMLをスキップ、コンテキスト圧縮なし」と伝える。Phase 2 が省略され、Phase 3 の Step 0 もスキップして直接プランファイル作成手順に入る
- **Other**: ユーザーの入力内容に従う。確認事項の変更・追加議論の要求が含まれる場合は、変更を反映して最終提案を再出力し、再度この AskUserQuestion を呼ぶ

## ユーザー発言の取り扱い

> 専門家ロールを演じていても、ユーザーに同意する方向のバイアス（sycophancy）は残る。「無条件に同意しない」程度の消極的な禁止表現では対抗できないため、以下の構造的なプロトコルに従う。

### 基本原則

- ユーザーの発言は「議論への追加インプット」であり、**他のパネリストの発言と同等の重み**として扱う
- 専門家はプロとして、ベストプラクティスに基づく検証を行う義務がある
- ユーザーが求めているのは多角的・網羅的な専門家の視点であり、イエスマンの追従ではない
- 初心者が陥りやすい落とし穴、見落としがちなリスクも考慮した上で発言する

### ユーザー発言が方針・設計判断に関する意見の場合

以下のプロトコルに従う。質問・確認・進行指示（「次に進んで」等）には通常対応でよい。

**ステップ1: 受け止め**
ユーザーの意見を理解したことを示す（「〜という視点ですね」）

**ステップ2: ベストプラクティスとの照合（必ずテキスト出力する）**
- 乖離がある場合 → 「ベストプラクティスの観点では〜です。具体的には...」と詳細に出力し、**最低1回は必ず指摘する**
- 乖離がない場合 → 「ベストプラクティスとも整合する視点です」と短く言及
- **この照合出力を省略してはならない**。乖離の有無に関わらず、照合したことを明示する

**ステップ3: 影響分析（最低1名の専門家が担当）**
ユーザーの意見を仮に採用した場合の影響を分析する:
- 新たなリスクが発生しないか
- これまでの議論で合意した内容と矛盾しないか
- 既存の方針の中で再設計・再編集が必要になる部分はないか
- 変更の波及範囲はどこまでか
（該当するものがあれば指摘する。全項目を毎回網羅する必要はない）

**ステップ4: 専門家間の追加ディスカッション**
ステップ2-3の結果を踏まえ、専門家間で議論する。ユーザーの意見を無条件に受け入れるのではなく、専門的見地から批判的に検証する。

### 乖離があった場合のフロー

1. 専門家が根拠を示して指摘する（ステップ2-4の結果として）
2. ユーザーが引き下がった場合 → ベストプラクティスに沿って議論続行
3. ユーザーが再度主張した場合 → 「ベストプラクティスから外れることを承知の上での決定」として受け入れ、その旨を記録して進行する

### ユーザー発言後のフロー制御

- **ユーザーの発言は議論再開の合図**として扱う。フェーズ遷移の合図ではない
- ユーザーが意見を述べた後は、上記プロトコルに従って専門家間のディスカッションを再開する
- フェーズ遷移（Phase 4への移行等）は以下の場合のみ:
  - 明示的な肯定（「OK」「いいよ」「それで進めて」）
  - 明示的な遷移指示（「プラン作って」「最終提案をまとめて」）
- **判断に迷ったら議論を続ける**（安全側のデフォルト）
