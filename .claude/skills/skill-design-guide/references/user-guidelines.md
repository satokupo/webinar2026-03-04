## インラインコード実行の禁止

どんなに簡単な処理であっても、`-c` や `-e` オプションでのインラインコード実行は使わない。

```
NG: python -c "print('hello')"
NG: bash -c "ls -la"
NG: node -e "console.log('test')"
```

必要な処理は必ず Python ファイル等を作成してスキルに内包する。

## .gitignore の操作ルール

### 原則

**リポジトリの .gitignore をスキルのために修正することは禁止。**

スキル内で追跡除外したいファイルがある場合は、スキルディレクトリ内に `.gitignore` を配置して対処する。

### 正しい方法と禁止されている方法

| やりたいこと | 正しい方法 | 配置場所 | 禁止されている方法 |
|---|---|---|---|
| スキル内ファイルの追跡除外 | スキルディレクトリ内に `.gitignore` を配置 | `claude/skills/<skill-name>/.gitignore` | リポジトリの `.gitignore` に除外ルールを追加 |
| スキル自体の追跡 | 何もしない（既に追跡対象） | — | リポジトリの `.gitignore` に追跡ルールを追加 |

### 理由

- スキルはグローバル設定であり、シンボリックリンクで全リポジトリに展開される
  → 特定リポジトリの `.gitignore` にルールを入れても、他のリポジトリでは効かない
  → だからスキル内に `.gitignore` を内包する必要がある
- dotfiles リポジトリの `.gitignore` は `!claude/skills/**` でスキル内を全追跡している
  → スキルの追跡には何の設定も不要（既にホワイトリストに含まれている）
  → 逆に、スキル内の全ファイルがGit管理されるため、除外にはスキル内 `.gitignore` が唯一の手段

### 具体例

**NG — リポジトリの .gitignore にスキルの除外ルールを追加:**
```
# dotfiles/.gitignore に以下を追加 ← これは禁止
claude/skills/my-skill/.env
claude/skills/my-skill/local-data/
```
このリポジトリでしか効かず、スキルが展開される他の全リポジトリでは除外されない。

**OK — スキルディレクトリ内に .gitignore を配置:**
```
# claude/skills/my-consigliere/.gitignore ← これが正しい
global-memories/
```
スキルと一緒にシンボリックリンクで展開されるため、全リポジトリで除外が効く。

## 標準ディレクトリと役割

スキルで使用するディレクトリをコアと拡張の2層に分類する。これ以外のディレクトリ名（helpers/, utils/, lib/ 等）は非推奨。

### コアディレクトリ（公式定義）

| ディレクトリ | 役割 | 配置するもの |
|---|---|---|
| `scripts/` | 実行コード | Python スクリプト、シェルスクリプト、ヘルパー、ユーティリティ |
| `references/` | 知識・手順 | ドメイン知識、実行フロー、ガイドライン、品質チェックリスト |
| `assets/` | 汎用素材 | 画像、フォント、アイコン、設定サンプル等（そのまま使う/コピーするもの） |

### 拡張ディレクトリ（ユーザー定義で許可）

| ディレクトリ | 役割 | 配置するもの |
|---|---|---|
| `templates/` | 出力の型 | 変数を埋めて使うひな型、コード生成テンプレート、出力フォーマット定義 |
| `docs/` | スキル固有の設計記録 | 当該スキル専用の設計判断・拡張方針（後述の制約あり） |
| `subagent-prompts/` | サブエージェントへの指示書 | Agent ツールでサブエージェントにパスで渡し、直接読んで実行させるプロンプト |

拡張ディレクトリは user-guidelines.md で明示的に許可されたもののみ使用可能。

### ルート直置きファイル

- **SKILL.md** — 必須
- **README.md** — 公開スキルのみ（ソフトウェア慣習に従いルートに配置）
- **.gitignore** — 必要時

### 使い分け判断基準

- **AIが理解して判断に使う情報** → `references/`
- **AIが出力時にそのまま使う/コピーする素材** → `assets/`
- **AIが変数を埋めて出力を生成する型** → `templates/`
- **サブエージェントがパスで受け取り、直接読んで実行する指示** → `subagent-prompts/`
- **スキルの通常動作には関わらない設計記録** → `docs/`

### docs/ ディレクトリの制約

docs/ はスキルの修正・拡張時にAIが参照するもので、通常運用では読み込まない。

**配置してよいもの**:
- 当該スキル専用のルール・方針（スキルデザインガイドの一般原則から逸脱する場合の記録）
- スキル拡張時に参照するサンプルや方針書（例: DESIGN.md）

**禁止**:
- ログ・履歴的なファイル（開発ログ、変更履歴等）。古い情報を正として扱うリスクがある
- 通常運用で参照するドキュメント（それは references/ に置く）

SKILL.md に docs/ への参照を書く場合は、「改修・拡張時のみ参照」という条件付きルーティングとする。

### 分割の目安

上記のディレクトリ構成で不足を感じたら、個別スキル内で独自ディレクトリを作るのではなく、スキル分割を検討するか、本ファイル（user-guidelines.md）への拡張ディレクトリ追加を提案する。

### Anthropic ファイルの命名ルール

anthropic-guidelines.md を分割する必要が生じた場合、分割後の両方のファイルに `anthropic-` プレフィックスを付ける（例: `anthropic-guidelines-core.md`, `anthropic-guidelines-examples.md`）。出典の明確性を維持するため。

## AI判断ベースの構造設計原則

### 設計思想

スキルはAI判断ベースで動作する。ソフトウェアのビルドシステムやインポート解決とは本質的に異なり、ファイルの読み込みはSKILL.mdの自然言語指示に基づくAIの判断で行われる。

ソフトウェア開発では関心の分離のためにディレクトリを細かく分けることが良い設計とされるが、スキルでは逆効果になり得る。AIはディレクトリ構造をインポート文として機械的に解決するのではなく、自然言語の指示を読んで判断する。分岐が多すぎると判断漏れのリスクが高まる。

**人間のための整理ではなく、AIのための整理を意識すること。**

### 実践ルール

1. **1ルーティング = 1判断**: SKILL.md のルーティング指示は、AIが1回の判断で「読む/読まない」を決められる粒度にする。「AかつBの条件でCを読む、ただしDは除く」のような複合条件は避け、フラットな when-then 分岐にする

2. **ディレクトリ深さは1階層まで**: `references/prompts/` のような1階層は許容するが、それ以上のネスト（`references/domain/sub-domain/`）は禁止。パス文字列が長くなるだけでAIの判断精度は上がらない

3. **ルーティング記述のコンパクトさ**: SKILL.md のルーティング部分が肥大化したら、スキル分割を検討する。ルーティングの複雑さはファイル数の多さではなく判断分岐の多さに起因する

4. **ファイル名の自己記述性**: AIは名前から内容を推測する。`ref-001.md` ではなく `aws-deploy-guide.md` のように、内容を端的に表す英語の kebab-case で命名する

## SKILL.md 最小化ルール

anthropic-guidelines.md の Progressive Disclosure を、ユーザーポリシーとしてより厳格に適用する。
公式の500行以内を大幅に厳格化する理由と根拠の詳細は `references/token-optimization.md` を参照。

### 行数制限（フロントマター抜き）

| 区分 | 行数 | 説明 |
|------|------|------|
| 推奨 | 50行 | 単純〜中程度のスキルの目安 |
| 許容 | 100行 | 多分岐スキルの上限目安 |
| 硬い上限 | 150行 | これを超える場合はreferences/への分離が不十分 |

理由: コンパクト後にSKILL.MDが全文強制リロードされ続けるため、行数を最小化する必要がある。
ただし行数を削るあまりルーティング指示が曖昧になり精度が落ちることは本末転倒。ルーティング精度を犠牲にするなら上限の緩和を検討する。

### ファイル分離の原則

SKILL.MDに残すもの:
- 概要（1-2文）
- 絶対制約（2-3行）
- ルーティング指示（条件に応じたreferences/への分岐）

実行詳細・手順・具体例はすべてreferences/に分離する。**分岐の有無を問わない統一ルール**。
分離先は `references/` ディレクトリに統一する（判断コストの排除）。

例外: フロントマター抜き20行以下の極小スキルは分離不要。

### ファイル読み込み制御の一元化

SKILL.MDがすべてのファイル読み込みの司令塔となる。
references/内のファイルに「次は○○を読め」という読み込み指示を書かない。

理由:
- LLMの指示追従信頼性: references内の読み込み指示は見落とされるリスクがある
- デバッグ容易性: SKILL.MDだけ見れば読み込みの全体像がわかる

### 1実行あたりのRead回数目安

メインフロー内で**2回以内**を基本とする。

ただしルーティング先の1セットとして常に併読するファイル群（例: 構造チェック時に同時に読む複数ファイル）は、1回のルーティングとしてカウントする。

## description の言語ルール

descriptionは全スキルのメタデータとして常時コンテキストに常駐する。トークン効率を最大化するため:

- **機能説明は英語で記述する**（日本語は同じ情報量でより多くのトークンを消費する）
- **トリガーキーワードのみ日本語OK**（ユーザーが日本語で発話するため、マッチングに必要）

```yaml
# 良い例
description: |
  Comprehensive knowledge for skill design, structure checking,
  and new skill creation.
  Trigger: 「スキルの構造をチェックして」「スキルを新しく作りたい」

# 悪い例（全文日本語）
description: |
  スキルの設計・構造チェック・新規作成時に参照するナレッジ集。
  スキルを作成・改善する際、構造のベストプラクティスを確認したい時、
  「スキルの構造をチェックして」「スキルを新しく作りたい」等の
  発言があった場合に参照すること。
```

## Trigger キーワードの品質基準

### 原則

Trigger はスキルの「外部インターフェース」であり、正しいスキルを一意に起動することが唯一の責務。
スキル内部のフェーズ遷移や進行制御は Trigger の責務外（内部ロジックで処理する）。

本ルールは Trigger キーワードの品質に関するものであり、description 本文の説明範囲を制限するものではない。
「少し強気に書く」（undertrigger 対策）との併用が前提。

### 一意性テスト

キーワード追加時に以下を自問する:

> 「このキーワード単体を聞いて、特定の1つのスキルが一意に想起されるか？」

答えが No なら、そのキーワードは Trigger に不適格。

### 禁止パターン

| パターン | 例 | 理由 |
|---|---|---|
| 汎用的な段階・進行表現 | 「フェーズN」「ステップN」「Phase N」「次に進んで」 | 複数スキルに内部フェーズが存在しうる |
| スキル内部フェーズの外部公開 | 「フェーズ5」（特定スキルの内部Phase起動意図） | 内部ロジックの外部漏出。カプセル化違反 |
| 一般的すぎる動作語単体 | 「確認して」「チェックして」「実行して」 | 多数のスキルに該当しうる |
| 他スキルの固有領域と重複 | 「CODEX検証」→ codex-review の領域 | 名前空間の衝突 |

### 良い Trigger の条件

- そのスキル固有の名称・用語を含む（例: 「専門家を召喚」「ログ保存して」）
- ユーザーが自然に発話するフレーズである
- 他スキルの Trigger と意味的に重複しない

### 既存スキルへの適用

新ルール策定以前のスキルにも段階的な適合を推奨する。

### スキルのアーカイブ

使わなくなったスキルは `skills/archive/` に移動する。archive/ 配下はスキル自動読み込みスコープ外となる。Git追跡はそのままでよい（参考資料として保持）。

## name フィールドとスラッシュコマンド名

name フィールドの値がそのままスラッシュコマンド名（`/name`）になる。
ディレクトリ名は管理用の識別子であり、コマンド名とは独立している。
`command` フロントマターフィールドは機能しないため使用しない。

user-invocable: true のスキルでは、ディレクトリ名をそのまま name にせず、
スラッシュコマンドとして直感的に呼び出せる短い名前を検討する。

| ディレクトリ名 | name | コマンド | 備考 |
|---|---|---|---|
| git-commit-rules | commit | /commit | 短縮して直感的に |
| save-chatlog | save-chatlog | /save-chatlog | そのままでも直感的 |

## スキル間連携ルール

他スキルの機能を利用したい場合の方針。外部スキルの内部構造へのハードコードを防ぎ、リファクタリング耐性を確保する。

### 優先順位

1. **第一選択: 機能のコピー＆最適化** — 必要な機能が小さく安定しているなら、自スキル内にコピーして最適化する。依存関係が生まれず、シンプル
2. **第二選択: Skillツールでスキル名指定呼び出し** — コピーでは非効率な場合（対象が大きい・変更頻度が高い・状態依存）、Skillツールでスキル名を指定して呼び出す
3. **禁止: パス指定でのSKILL.MD Read** — SKILL.MDがルーティング専用になったため、Readで読んでも「references/を読め」が返るだけで機能しない。パス記述自体を禁止する

### フロントマター設定

- 呼ばれる側のユーティリティスキルは `user-invocable: false` を設定する（AIからSkillツールで呼び出し可能、ユーザーメニューには表示されない）
- `disable-model-invocation: true` のスキルはAIからも呼べないため、他スキルから呼ばれるスキルには設定しない

### 補足

- 粒度が合わない（大きなスキルの一部だけ欲しい）場合は、その機能の独立スキル化を検討する
- スキルが**生成した成果物**（`.claude/temp/` や `.claude/plans/` 内のファイル）への参照はこのルールの対象外。スキルの内部構造ではなく出力なので別レイヤー

### 具体例

**良い例:**
- 他スキルの一部の小さなロジックだけが必要な場合、その部分を自スキル内にコピーして使う
- Skillツールで `<utility-skill>` を呼び出す（スキル名指定）
- スキルが出力した成果物を別スキルのインプットとして使う（内部構造への依存ではなく、出力の参照）

**悪い例:**
- `.claude/skills/xxx/SKILL.md` をReadで読む指示を記載（SKILL.MDパス指定のRead）
- `.claude/skills/planmode-rules/references/validation-flow.md` を Read で読む指示を記載（内部ファイルへのハードコード）
- `python3 .claude/skills/他スキル/scripts/xxx.py` を直接実行する指示を記載

## context: fork ガイドライン

フロントマターの `context: fork` を使うと、スキルがサブエージェント（別プロセス）として実行される。

### 使うべきケース

- 自己完結した重い処理（大量ファイルの一括処理等）
- 会話文脈が不要な独立タスク
- メインの会話コンテキストを圧迫したくない場合

### 使わないケース

- 会話文脈が必要な処理（ユーザーとのインタラクティブなやりとり）
- 軽量な処理（forkのオーバーヘッドの方が大きくなる）
- メインコンテキストの情報を参照する必要がある場合

### 注意事項

- `$ARGUMENTS` 等の変数置換はfork先でも動作する（プリプロセッサがfork前に置換するため）
- ただし会話履歴はfork先から参照できない
- `agent` フィールドでエージェントタイプを指定可能（Explore, Plan, general-purpose, カスタム）

## 完了時品質チェック

### 推奨対象

以下のいずれかに該当するスキルには、品質チェック機構の導入を推奨する:

- 複数ファイルの操作を伴うスキル
- 副作用のある変更（ドキュメント更新、設定ファイル変更等）を含むスキル
- 作業完了後に別の場所への反映が必要なスキル
- 外部に影響を与える出力を生成するスキル（メール送信、API呼び出し、ファイル公開等）

単一ファイルのみの単純操作スキルには不要。

### 方式

「静的ベース＋動的拡張」方式を推奨する:

- **静的部分**: スキル固有の必須チェック項目を独立ファイルに定義（**標準ファイル名: `references/quality-checklist.md`**）
- **動的部分**: タスク開始時にLLMが影響範囲を宣言し、完了時にそれを検証

### 配置と組み込み

- チェック項目は独立ファイルとして配置する（推奨: `references/quality-checklist.md`）
- SKILL.md には2箇所に発動指示を記述する:
  1. フロー冒頭: 品質チェックの存在を予告し、影響範囲の宣言を指示
  2. フロー最終ステップ: チェックリストファイルの読み込みとチェック実行を指示
- 2箇所に書く理由: LLMの注意維持。冒頭だけ/末尾だけでは飛ばされるリスクがある

### エビデンス付きチェック

チェック結果は「OK」だけでなく、1項目1行で根拠を記載する:

````
- [x] 変更ファイル一覧を提示: config.ts, api.ts の2ファイルを変更
- [x] 関連ドキュメント更新: CLAUDE.md の構成一覧にエントリ追加済み
- [N/A] テスト実行: 本変更はドキュメントのみのため該当なし
````

チェック不合格時は修正→再チェックのループを全項目パスまで繰り返す。
チェック結果はタスク完了報告の一部としてユーザーに提示する。
