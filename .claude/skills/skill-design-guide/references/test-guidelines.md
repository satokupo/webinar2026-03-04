# 軽量テストガイドライン

スキルの作成・編集時に、テストの要否を判定する基準。
テストはスキル内に組み込まず、プラン作成時にテスト工程として追加する（メタレベル）。

## リスクカテゴリ

### カテゴリ1: 実行順序リスク
- **条件**: 複数のコマンド/ステップを特定の順序で実行する
- **例**: Codex ヘッドレス呼び出し、パイプライン実行
- **テスト内容**: 手順を1回通しで実行し、正常終了を確認

### カテゴリ2: 入力バリエーションリスク
- **条件**: 異なる入力パターンで異なる挙動をする（条件分岐、ルーティング）
- **例**: フェーズ別ルーティング、モデル選択
- **テスト内容**: 主要パターン2〜3通りで実行

### カテゴリ3: 外部依存リスク
- **条件**: 外部API、CLIツール、環境変数に依存する
- **例**: APIキーの存在チェック、コマンドのパスが通っているか
- **テスト内容**: 依存の存在確認 + 基本的な疎通確認

### カテゴリ4: 出力フォーマットリスク
- **条件**: 特定のファイル形式やディレクトリ構造を出力する
- **例**: YAML frontmatter 生成、ファイル保存先の構造
- **テスト内容**: 出力が期待するフォーマットか確認

### カテゴリ5: 破壊的操作リスク
- **条件**: ファイルの上書き、削除、外部サービスへの送信を含む
- **例**: /clear 送信、ファイル変換の上書き
- **テスト内容**: テスト用一時ファイルで dry-run

## テスト推奨マトリクス

| リスクカテゴリ | 新規作成時 | 既存編集時 |
|---|---|---|
| 1. 実行順序 | 推奨 | ロジック変更時のみ |
| 2. 入力バリエーション | 推奨 | 分岐追加/変更時のみ |
| 3. 外部依存 | 推奨 | 依存先変更時のみ |
| 4. 出力フォーマット | 任意 | フォーマット変更時のみ |
| 5. 破壊的操作 | 強く推奨 | 対象変更時のみ |

- **推奨**: AI がユーザーに「テストを追加しましょうか？」と提案する
- **強く推奨**: 理由を明示して強めに提案する
- **任意**: ユーザーが望めば実行

## 運用ルール

- テストはスキル内に組み込まない
- プラン作成時（planmode-rules 連携）にリスクカテゴリを判定し、該当すればプランにテスト工程を追加
- テスト工程の内容は AI がリスクカテゴリに応じて生成する（テンプレート固定にしない）
- これは skill-creator のフル eval とは別物。ここでカバーできない品質追い込みは SKILL.md の skill-creator 発動判定を参照
