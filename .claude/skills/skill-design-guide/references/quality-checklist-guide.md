# 品質チェック機構 実装ガイド

複数ファイルにまたがる複雑なタスクでは、作業完了後にやるべきことの漏れが発生しやすい。品質チェック機構はこの問題を構造的に解決するための仕組みである。

---

## 推奨ファイル構造

各スキルは以下の構造を持つことを推奨する:

```
skill-name/
├── SKILL.md
└── references/
    ├── quality-checklist.md  ← チェック項目を定義
    └── ...（その他の references）
```

---

## 静的ベース＋動的拡張方式

品質チェックは「静的ベース＋動的拡張」の2層で構成する。

### 静的部分

スキル固有の必須チェック項目を `references/quality-checklist.md` に定義する。これは「やったことの確認」に強い。フローのステップに対応する形で、完了時に必ず検証すべき項目を列挙する。

### 動的部分

タスク開始時にLLMが影響範囲を宣言し、完了時にその宣言に基づいて検証する。これは「やるべきことの発見」に強い。

> QAエンジニア「静的は『やったことの確認』に強く、動的は『やるべきことの発見』に強い。ユーザーの問題は『そもそも忘れた』なので静的だけでは不十分」

> プロンプトエンジニア「作業前に影響範囲を宣言させることで、作業中のトンネルビジョンを補正できる」

### 途中チェックを挟まない理由

チェックはタスク開始時（宣言）と完了時（検証）の2回のみ。途中チェックは挟まない。コンテキスト消費と作業効率のバランスを考慮した設計であり、途中で何度もチェックを実行するとコンテキストウィンドウを圧迫し、本来の作業品質が低下する。

---

## SKILL.md への組み込みパターン

品質チェックの発動指示は SKILL.md の **2箇所** に記述する。

### フロー冒頭の記載例

```markdown
**重要**: この作業には完了時品質チェックがある。作業開始前に変更対象ファイル・影響を受けるドキュメント・副作用の発生箇所を宣言すること。
```

### フロー最終ステップの記載例

```markdown
`references/quality-checklist.md` を読み込み、チェックを実行する。チェック不合格の場合は修正→再チェックを全項目パスまで繰り返す。
```

### 2箇所に書く理由

冒頭の予告と最終ステップの発動指示の両方が必要。

> プロンプトエンジニア「フローの冒頭だけ/末尾だけでは飛ばされるリスクがある。予告があることでLLMが最終ステップの存在を意識し続ける」

冒頭で予告することにより、LLMは作業中も「最後にチェックがある」ことを意識し続ける。最終ステップにのみ記載した場合、長い作業フローの途中でその存在が忘れられるリスクがある。

---

## エビデンス付きチェックの書き方

チェック結果は「何を確認して結果がどうだったか」を1項目1行で記載する。

```
- [x] 変更ファイル一覧を提示: config.ts, api.ts の2ファイルを変更
- [x] 関連ドキュメント更新: CLAUDE.md の構成一覧にエントリ追加済み
- [N/A] テスト実行: 本変更はドキュメントのみのため該当なし
```

- 「OK」だけの記載は不可。必ず根拠を添える
- 該当しない項目は「N/A: 理由」と明記する（チェックせずにスキップしたのか、該当しないのかを区別する）

> DevOpsエンジニア「LLMは全部OKとして流す傾向がある」

> プロンプトエンジニア「何を確認して結果がどうだったかを出力させることで対策する」

---

## チェック不合格時のフロー

チェック不合格（漏れ検出）時は以下のループを回す:

1. 漏れを検出する
2. 修正を実施する
3. 再チェックを実行する
4. 全項目パスするまで 1〜3 を繰り返す

チェックして漏れがありましたで終わったら意味がない。必ず修正までを含めたループとして設計する。

---

## チェック結果の出力

チェック結果はタスク完了報告の一部として、必ずユーザーに提示する。ユーザーの目に触れることが最大の品質保証である。

内部的にチェックを実行しても、その結果がユーザーに見えなければ形骸化のリスクが高い。チェック結果を明示的に出力することで、LLMの自己申告に対する透明性を確保する。

---

## quality-checklist.md の書き方ガイド

### 基本原則

- 箇条書き中心で記述する（1項目1行）
- SKILL.md のフローのステップと対応関係を持たせる
- ファイル冒頭に同期メモを入れる: 「SKILL.md のフローを変更した場合は本ファイルも確認すること」

### 汎用的なチェック項目の例

```
> SKILL.md のフローを変更した場合は本ファイルも確認すること。

## 変更の完全性
- [ ] 変更対象ファイルの一覧を提示したか
- [ ] 各ファイルの変更が意図通りか確認したか

## 参照整合性
- [ ] ファイル名変更時、参照元をすべて更新したか
- [ ] 新規ファイル作成時、SKILL.md から参照導線があるか

## 副作用の確認
- [ ] 関連ドキュメントの更新が必要か確認したか
- [ ] 設定ファイルへの反映が必要か確認したか

## 完了報告
- [ ] チェック結果をエビデンス付きでユーザーに提示したか
```
