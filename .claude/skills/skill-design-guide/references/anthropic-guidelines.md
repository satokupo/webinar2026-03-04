## スキルの構造

### ディレクトリ構成パターン

```
skill-name/
├── SKILL.md (必須)
│   ├── YAML フロントマター（name, description 必須）
│   └── Markdown による指示内容
└── Bundled Resources (任意)
    ├── scripts/    - 決定的・反復的タスク用の実行コード
    ├── references/ - 必要時にコンテキストに読み込むドキュメント
    └── assets/     - 出力に使うファイル（テンプレート、アイコン、フォント等）
```

### ディレクトリの使い分け

| ディレクトリ | 用途 | 読み込みタイミング |
|---|---|---|
| `scripts/` | Python 等の実行スクリプト | 実行時（コンテキストに読み込まず実行可能） |
| `references/` | 補足ドキュメント、ドメイン知識 | SKILL.md からの参照指示に従い必要時に読み込み |
| `assets/` | テンプレートファイル、画像等 | 出力生成時にコピー・参照 |

## Progressive Disclosure（段階的開示）

スキルは3層のローディングシステムを使う:

### Layer 1: メタデータ（常時コンテキスト内）

- name + description（約100語）
- スキルの発動判定に使われる
- **ここが最も重要** — description の書き方でスキルが発動するかどうかが決まる

### Layer 2: SKILL.md 本文（スキル発動時にコンテキスト内）

- **500行以内**を目安にする（理想は200〜300行）
- 500行を超えそうな場合は、階層を追加して参照先への分岐を設ける
- 常時必要な情報のみをここに書く

### Layer 3: バンドルリソース（必要時のみ）

- scripts/ は実行時にコンテキストに読み込まず実行できる
- references/ は SKILL.md の指示に従って必要時に読み込む
- サイズ制限なし（ただし参照ファイルは適切な長さに保つ）

### なぜ段階的開示が重要か

スキルのメタデータ（description）は**常にコンテキストに存在**する。
スキルが発動すると SKILL.md 本文がコンテキストに追加される。
SKILL.md が大きすぎると、スキル起動だけでコンテキストを大量消費し、
本来の作業に使えるコンテキストが減ってしまう。

そのため:
- SKILL.md には「ルーティング」と「常時必要な指示」だけを書く
- 詳細は references/ に分離し、トリガー発生時にのみ読み込む

## フロントマターの書き方

### 必須フィールド

```yaml
---
name: skill-name          # スキル識別子（ケバブケース）
description: |            # スキルの説明（発動トリガーとして機能）
  このスキルが何をするか、いつ使うべきかを記述。
---
```

### オプションフィールド

```yaml
user-invocable: false          # ユーザーが /コマンド で直接呼び出す場合のみ true にする
disable-model-invocation: true # AI自発呼び出しを禁止（ユーザー手動のみ）
argument-hint: "<query>"       # オートコンプリート時のヒント表示
context: fork                  # サブエージェント実行（メインコンテキストと分離）
agent: general-purpose         # context: fork 時のエージェントタイプ指定
                               # 選択肢: Explore, Plan, general-purpose, カスタム
allowed-tools:                 # スキル実行時のツール権限制御
  - Read
  - Grep
  - Glob
model: haiku                   # スキル実行時のモデル指定（sonnet, opus, haiku）
hooks:                         # スキルライフサイクルフック
  pre: "echo 'before skill'"
  post: "echo 'after skill'"
```

### 呼び出し制御

| 設定 | ユーザーが呼べる | AIが呼べる | description常駐 |
|------|:---:|:---:|:---:|
| デフォルト | Yes | Yes | Yes |
| `disable-model-invocation: true` | Yes | No | No |
| `user-invocable: false` | No | Yes | Yes |

### description の書き方のコツ

> **適用スコープ**: 以下のガイダンスは description を新規作成する場合、
> または description の改善を依頼された場合にのみ適用する。
> 既存スキルの構造チェック時に「この description は pushy か？」等の評価を毎回行う必要はない。

description はスキルの発動判定に使われる最も重要なフィールド。
Claude は description を見て「このスキルを使うべきか」を判断する。

**「少し強気に」書く（undertrigger 対策）**

Claude はスキルを「使わなさすぎる」傾向がある。
description には、スキルが対応するシナリオを幅広く明記する。

悪い例:
```yaml
description: ダッシュボードを作成するスキル。
```

良い例:
```yaml
description: |
  シンプルで高速なダッシュボードを構築するスキル。
  ダッシュボード、データ可視化、内部メトリクス表示、
  データの表示に関する発言があった場合に使用すること。
```

**トリガーフレーズを含める**

ユーザーがどんな言い方をした時にこのスキルが発動すべきかを具体的に書く。

```yaml
description: |
  スキルの設計・構造チェック・新規作成時に参照するナレッジ集。
  「スキルの構造をチェックして」「スキルを新しく作りたい」等の
  発言があった場合に参照すること。
```

**description の評価・改善時の知見**

スコープ: description を新規作成する場合、または description の改善を依頼された場合にのみ参照する。
既存スキルの構造チェック時に毎回評価する必要はない。

- **should-trigger / should-not-trigger の両面で考える**: スキルが発動すべきクエリと、発動すべきでないクエリの両方を想定する
- **near-miss が最も重要**: should-not-trigger で最も価値があるのは、キーワードが重なるが実は別の用途であるクエリ（明らかに無関係なクエリはテストとして無意味）
- **現実的なクエリで評価する**: 抽象的な「データを分析して」ではなく、具体的なファイル名・背景・口語表現を含むクエリを想定する
- **簡単なタスクではスキルは発火しない**: Claude は自力で処理できる単純なタスクにはスキルを参照しない。description がマッチしても、タスクが単純すぎると発火しないことがある

## 変数置換

SKILL.md内でプリプロセッサが置換する文字列変数。スキル発動時に実際の値に置き換えられ、Claudeがコンテキストとして保持する。

### 引数アクセス

```yaml
# ユーザーが /skill-name arg1 arg2 と入力した場合
$ARGUMENTS     # 全引数（"arg1 arg2"）
$ARGUMENTS[0]  # 第1引数（"arg1"）
$ARGUMENTS[1]  # 第2引数（"arg2"）
$1, $2         # $ARGUMENTS[0], $ARGUMENTS[1] の短縮形
```

## 動的コンテキスト注入

`` !`command` `` 構文を使うと、シェルコマンドの実行結果をスキル内容に事前埋め込みできる。

```markdown
Today's date is !`date +%Y-%m-%d`
Current branch: !`git branch --show-current`
```

### 注意事項

- スキル発動時にコマンドが実行され、結果がスキル本文に埋め込まれる
- コマンドの実行はスキルのロード時（references読み込み前）に行われる
- 長時間かかるコマンドはスキルの起動を遅延させるため避ける
- セキュリティ: 信頼できるコマンドのみ使用する

## description budget

全スキルのdescriptionは常時コンテキストウィンドウに存在し、budgetで管理される。

- **デフォルト**: コンテキストウィンドウの2%（フォールバック: 16,000文字）
- **上書き**: 環境変数 `SLASH_COMMAND_TOOL_CHAR_BUDGET` で変更可能
- スキル数が多すぎてbudgetを超過すると、一部スキルのdescriptionが除外される
- `disable-model-invocation: true` のスキルはdescription常駐から除外される

## SKILL.md の書き方

### Writing Patterns

**命令形を使う**

指示は命令形で書く。「〜してください」ではなく「〜する」「〜せよ」。

**出力フォーマットを定義する**

期待する出力形式がある場合は、テンプレートとして明示する:

```markdown
## レポート構造
必ず以下のテンプレートを使用:
# [タイトル]
## エグゼクティブサマリー
## 主な発見事項
## 推奨事項
```

**例を含める**

具体例は理解を助ける。Input/Output 形式で書くとわかりやすい:

```markdown
## コミットメッセージのフォーマット
**例 1:**
Input: JWT トークンによるユーザー認証を追加
Output: feat(auth): JWT ベースの認証を実装
```

### 書き方のスタイル

> **適用スコープ**: 以下のスタイルガイダンスは SKILL.md を新規作成する場合にのみ適用する。
> 既存スキルのチェック時に MUST/NEVER の使用を毎回指摘する必要はない。
> セーフティクリティカルな箇所（省略するとバグる、ユーザーに被害がある場面）では
> 強い制約表現も正当である。

- **WHY を説明する**: 重苦しい MUST/NEVER の連発より、なぜそうすべきかの理由を書く方が効果的
- **一般化する**: 特定の例にだけ合うような狭い指示は避け、幅広く適用できる原則を書く
- **簡潔に保つ**: 機能していない指示は削除する。指示が長いほどモデルの注意が分散する
- **理論的背景を伝える**: 今の LLM は賢い。理由がわかれば、具体的な指示がなくても適切に判断できる

### ドメイン別 references 分離パターン

スキルが複数のドメインやフレームワークに対応する場合、
ドメインごとに reference ファイルを分離する:

```
cloud-deploy/
├── SKILL.md (ワークフロー + ドメイン選択のルーティング)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

SKILL.md でユーザーの要件に応じたドメインを判定し、
該当する reference ファイルだけを読み込む。
全ドメインを一度に読み込まないことでコンテキストを節約する。

## Reference ファイルのガイドライン

- **300行を超えたら目次（TOC）を付与する** — ファイル冒頭に目次を置き、必要なセクションだけを参照できるようにする
- SKILL.md から「いつ・どのファイルを読むか」を明確に指示する
- 1ファイルが長大になりすぎたら、ドメインや用途で分割を検討する
- reference ファイルは「AI エージェントが参照するもの」であり、人間向けドキュメントではない

---

## 含めないもの

以下のファイルはスキルに含めない:

- **README.md** — スキルは AI エージェント向け。人間向けオンボーディングは不要
- **INSTALLATION_GUIDE.md** — 同上
- **CHANGELOG.md** — バージョン管理は git で行う
- **その他の補助ドキュメント** — スキルの動作に必要ないファイルは含めない

## 構造チェックリスト

既存スキルをチェックする際の観点:

### フロントマター

- [ ] `name` が設定されているか
- [ ] `description` が設定されているか
- [ ] `description` に発動トリガーとなるフレーズが含まれているか
- [ ] `description` が「少し強気」に書かれているか（undertrigger 対策）
- [ ] `description` が英語ベースで書かれているか
- [ ] 日本語はトリガーキーワードの引用のみか
- [ ] 公式に存在しないフロントマターキーが含まれていないか

### サイズと構造

- [ ] SKILL.md がuser-policiesの行数制限に従っているか（具体的数値はuser-policies参照）
- [ ] 実行詳細がreferences/に分離されているか（概要・絶対制約・ルーティング以外）
- [ ] 常時必要な情報だけが SKILL.md にあるか（作業分岐は別ファイルへ）
- [ ] 不要なファイル（README.md, CHANGELOG.md 等）が含まれていないか

### Progressive Disclosure

- [ ] SKILL.md はルーティングと常時必要な指示に絞られているか
- [ ] references/ ファイルへの参照指示が明確か（いつ・何を読むか）
- [ ] 300行超の reference ファイルに目次があるか
- [ ] references内のファイルに読み込み指示（「次は○○を読め」）が含まれていないか

### 書き方

- [ ] 指示が命令形で書かれているか
- [ ] WHY（なぜそうすべきか）が説明されているか
- [ ] 出力フォーマットが定義されているか（該当する場合）
- [ ] 例が含まれているか（該当する場合）
- [ ] 不要な MUST/NEVER の乱用がないか

### ドメイン分離

- [ ] 複数ドメイン対応の場合、reference ファイルが分離されているか
- [ ] 全ドメインを一度に読み込まない設計になっているか

### スキル間連携

- [ ] 他スキルの内部ファイル（SKILL.md 以外）を直接参照していないか
- [ ] パス指定でのSKILL.MD Read方式を使っていないか（Skillツールでスキル名指定を推奨）
- [ ] 呼ばれる側のスキルに適切なフロントマター設定があるか（`user-invocable: false` 等）

### 品質チェック設計

- [ ] 品質チェックの要否を判断したか（複数ファイル操作・副作用・外部影響のいずれかに該当するか）
- [ ] 必要な場合、`references/quality-checklist.md` が存在するか
- [ ] SKILL.md の2箇所（フロー冒頭の予告＋最終ステップのチェック指示）に配置されているか

### AI判断への配慮

- [ ] ルーティング指示が複合条件（AかつBかつC）になっていないか
- [ ] ディレクトリの深さが1階層以内に収まっているか
- [ ] ファイル名が内容を端的に表しているか（ref-001.md のような連番命名でないか）

## SKILL.md フォーマット例

新規スキル作成時の参考として、推奨構成のサンプルを示す。
これはそのままコピーして使うテンプレートではなく、構成の参考例。

```markdown
---
name: example-skill
description: |
  このスキルが何をするかの説明。
  「○○して」「△△したい」等の発言で起動する。
user-invocable: false
---

# スキル名

スキルの概要を1〜2文で説明。

## トリガー条件

どのような場面・発言でこのスキルが使われるかを記載。

## 処理フロー

メインの処理手順を記載。
分岐がある場合は、どの条件でどのファイルを参照するかを明記。

### ケース A の場合

→ `references/case-a.md` を読み込んで対応

### ケース B の場合

→ `references/case-b.md` を読み込んで対応

## 出力フォーマット

期待する出力の形式がある場合はここで定義。

## 注意事項

守るべきルールや制約があれば記載。
```

上記の構成ポイント:
- フロントマターで発動条件を明確に定義
- 本文はルーティング中心で、詳細は references/ に委譲
- 処理フローが一目でわかる構成
- 必要なセクションだけを含む（不要なセクションは省略）
