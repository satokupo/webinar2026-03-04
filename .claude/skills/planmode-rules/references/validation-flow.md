## トリガー

- **初回検証**（「検証準備を行って」「検証準備して検証して」等、「検証準備」を含む表現）:
  前処理スクリプトを実行し（`.claude/temp/` に会話ログ作成）、検証エージェント3つ（Opus+Sonnet+Haiku）を並列起動する。
  初回検証からそのまま自動再帰ループに入り、収束または中断まで回し続ける。
- **再開**（「完璧ですか？」「これで完璧？」）:
  中断後の再開トリガー。
  保持しているエージェントIDでresumeし、自動再帰ループを再開する。

Phase 3完了後に自動では発動しない。

### 条件付き観点の起動判定

常設6観点に加え、以下のルーティングテーブルに基づいて条件付き観点を追加する。
テーブルの「起動条件（要約）」で一次スクリーニングし、該当するファイルをReadして詳細を確認する。
迷った場合は**起動する側に倒す**（不要に起動してもコスト増のみ、起動漏れはリスク）。

| ラベル | 起動条件（要約） | ファイル |
|--------|----------------|---------|
| スキル構造検証 | skills/配下の変更、SKILL.md作成・編集 | `conditions/skill-structure.md` |
| issueファイル保護 | issues/配下の操作、issue解決 | `conditions/issue-protection.md` |
| リポジトリ外書き込み | /tmp等リポジトリ外へのファイル操作 | `conditions/external-write.md` |

### Haiku フォーマットチェックエージェント

常設観点6（テンプレートフォーマット準拠）を担当する軽量エージェント。

> **なぜ resume なし・エージェントID保持なしなのか**:
> 観点6はステートレスな構造比較タスク（テンプレートとプランを読んで一致を確認するだけ）であり、
> 対話的なフィードバックループのコンテキストを保持する必要がない。
> 毎ラウンド新規起動しても結果は同一なため、resume によるコンテキスト維持は不要。
> Opus/Sonnet の検証エージェントが resume を使うのは、指摘→修正→再確認の対話コンテキストが
> 品質に影響するためであり、フォーマットチェックにはこの要件がない。

## 検証の核心

このフェーズの目的は**表面的なチェックではなく、プランの本質的な完成度を見抜く**こと。各項目を形式的に「OK」と流さず、実際にプランを受け取る側（＝別セッションのAI、または未来の自分）の視点で厳しく読むこと。

## 検証手順

### Step 1: 前処理スクリプト実行

```bash
python3 .claude/skills/planmode-rules/extract_conversation.py \
  --session-id <プランのフロントマターから取得> --cwd "$(pwd)"
```

- セッションIDはプランファイルのフロントマター `planning-session:` から取得
- 出力: `.claude/temp/{session_id}_conversation.md`（プロジェクトローカル）
- 前処理は**1回だけ実行**する

### Step 2: モデル割り当て決定

| トリガー表現 | 検証Opus側エージェントのモデル |
|-------------|----------------------|
| 「検証準備」（デフォルト） | opus |
| 「Sonnetで検証準備」 | sonnet |

※ 旧トリガー「Opusで検証準備」「しっかり検証して」はデフォルトが Opus になったため**廃止**。これらの表現が使われた場合もデフォルト（opus）として扱う。

※ 検証Sonnet側エージェントのモデルは sonnet 固定であり、上記テーブルの選択には従わない。

### Step 3: 検証エージェント3並列起動

Task ツールで**3個の検証エージェント（Opus側・Sonnet側・Haiku側）を並列起動**する。3つのTask呼び出しを**1回のレスポンス内で同時に**発行する（`run_in_background` は使わない）。

**重要**: 前処理で抽出される会話ログにはtool call結果（ファイル内容・プラン本文等）が含まれない。検証エージェントがプラン内容と発言を突き合わせられるよう、プランファイルを会話ログとは**別に**渡す。

※ `.claude/temp/` はプロジェクトローカルディレクトリ。起動時は `{session_id}` を実際の値に置換する

#### Opus側エージェント（会話ニュアンス+判断力観点）

- **モデル**: Step 2 で決定したモデル（デフォルト opus）
- **担当観点**: 観点1（会話ニュアンスの反映）、観点2（新規セッション実行可能性）、観点5（サブエージェント設計の妥当性）、条件付き観点（該当時）
- **入力**: conversation.md + プランファイル
- **プロンプト**: 下記Opus側テンプレートを使用。担当する観点（1,2,5）のチェック基準と、観点1への追加指示を含める。条件付き観点は、本ファイルの「条件付き観点の起動判定」セクションのルーティングテーブルに従い、該当時はOpus側のプロンプトに含める（Sonnet側には含めない）
- **エージェントID変数名**: 「検証Opus ID」として保持
- **注意**: この観点配分はユーザーの「迷うならOpus側」方針に基づく。Sonnet側への観点移動は禁止

#### Sonnet側エージェント（機械的チェック観点）

- **モデル**: sonnet（固定。Step 2 のモデル選択には従わない）
- **担当観点**: 観点3（指示の具体性）、観点4（影響範囲の網羅性）
- **入力**: プランファイルのみ（conversation.mdは渡さない）
- **プロンプト**: 下記Sonnet側テンプレートを使用。担当する観点（3,4）のチェック基準のみを含める。conversation.md関連の指示は一切含めない。条件付き観点は含めない
- **エージェントID変数名**: 「検証Sonnet ID」として保持

#### Haiku側エージェント（フォーマットチェック観点）

- **モデル**: haiku（固定）
- **担当観点**: 観点6（テンプレートフォーマット準拠）
- **入力**: plan-skeleton.md + プランファイル
- **プロンプト**: 下記Haiku側テンプレートを使用
- **resume**: 不要。毎ラウンド新規起動する（理由は「Haiku フォーマットチェックエージェント」セクション参照）
- **エージェントID保持**: 不要

#### 起動後の出力

3エージェント起動後、自動ループ開始前に以下を出力する:

```
🚀 自動検証を開始します
```

#### Task呼び出し方法

各エージェント（Opus側・Sonnet側・Haiku側）とも、テンプレートファイルのパスと入力ファイル情報をpromptで渡す。
固定指示はテンプレート内にあるため、promptは短い参照情報のみで構成する。

**Opus側 Task prompt**:
```
以下のテンプレートファイルを読み込み、指示に従って検証を実行してください。

テンプレート: .claude/skills/planmode-rules/subagent-prompts/opus-validation.md

入力ファイル:
- 構造化会話ログ: .claude/temp/{session_id}_conversation.md
- プランファイル: {plan_file_path}

上記ファイルをすべて読み込んでから検証を開始してください。
```

条件付き観点が該当する場合、該当する各条件付き観点ファイルごとに以下を追記（複数該当時は複数行列挙）:
```
追加チェック: {conditions_file_path} の「検証プロンプト」セクションに従い、追加観点も検証してください。
```

**Sonnet側 Task prompt**:
```
以下のテンプレートファイルを読み込み、指示に従って検証を実行してください。

テンプレート: .claude/skills/planmode-rules/subagent-prompts/sonnet-validation.md

入力ファイル:
- プランファイル: {plan_file_path}

上記ファイルを読み込んでから検証を開始してください。
```

**Haiku側 Task prompt**:
```
以下のテンプレートファイルを読み込み、指示に従って検証を実行してください。

テンプレート: .claude/skills/planmode-rules/subagent-prompts/haiku-format-check.md

入力ファイル:
- フォーマットテンプレート: .claude/skills/planmode-rules/templates/plan-skeleton.md
- プランファイル: {plan_file_path}

上記ファイルをすべて読み込んでから検証を開始してください。
```

### ラウンドごとのステータス出力

各ラウンドの Step 4 に入る前に、テーブル形式でステータスを出力する:

```
🔄 ラウンド N

| 観点 | 担当 | 結果 | 備考 |
|------|------|------|------|
| 1. 会話ニュアンス | Opus | ✅ OK | — |
| 2. 実行可能性 | Opus | ⚠️ 要修正 | 〜が不足 |
| 3. 指示の具体性 | Sonnet | ✅ OK | — |
| 4. 影響範囲の網羅性 | Sonnet | ✅ OK | — |
| 5. サブエージェント設計 | Opus | ✅ OK | — |
| 6. フォーマット準拠 | Haiku | ✅ OK | — |
```
（※ 該当する条件付き観点があれば `[条件付き] ラベル名 | Opus | ...` の行を追加）

- `🔄 ラウンド N` ヘッダーで現在のラウンド数を明示
- Step 8 と同じテーブルフォーマット（✅/⚠️ 絵文字で視覚的に明確化）
- 要修正の観点には備考欄に概要を記載
- 「担当」列でどちらのエージェントが報告した観点かを明示

### Step 4: メインOpusが評価・フィードバック作成

3検証エージェントの報告を読み（3エージェントの結果が揃ってから評価を開始）、以下のガイドラインに基づいて各提案を採用/却下する:

**採用の方向で判断する場合**:
- ユーザーの発言・意図と一致する指摘
- プランの実行可能性・明確性を向上させる提案
- 既知の問題（曖昧な表現、未定義の判断基準等）を解消する指摘

**却下の方向で判断する場合**:
- ユーザーの意図と矛盾する提案
- 過剰な制約を追加し実装の柔軟性を損なう提案
- 形式的な指摘に留まり実質的な改善につながらないもの

条件付き観点の報告を検証する際、指摘内容に疑問がある場合はメインAI自身も該当する条件付き観点ファイルの「検証内容」セクションに記載の参照先を確認して照合してよい。

#### 横断チェック

Opus側の報告とSonnet側の報告の間に矛盾がないか確認する:
- 例: Sonnet側（観点4）が「影響範囲に漏れがある」と報告した場合、Opus側（観点2）が「実行可能性OK」としている前提と矛盾しないか
- 矛盾が見つかった場合は、メインがプラン修正を行い、該当する側のエージェントに再確認をresumeで依頼する。矛盾が両エージェントの報告に関わる場合は、プラン修正後に両方のエージェントにresumeで再確認を依頼する

全報告の「確認内容と判断根拠」を横断的に照合し、ある観点でOKとした前提が別の観点の報告内容と矛盾していないか確認する（例: 観点3で「具体性OK」としているが、観点4の影響範囲調査で漏れが示唆されているケース等）。

判断結果を**構造化フィードバック**として作成し、プランの修正も行う。各エージェント向けに個別のフィードバックを作成する。修正が不要な側にはフィードバックを送らない（resumeしない）。

**フィードバックテンプレート**:

```
## フィードバック

### 各提案への判断
- 観点N・提案内容: 採用 / 却下
  - 理由: ...
  - （採用の場合）プランの○○セクションに反映済み

### プラン差分サマリー
- 変更1: ○○を追加
- 変更2: △△を修正

### 追加チェック依頼
- （あれば）○○の観点で再確認してほしい
- （なければ）追加チェックなし
```

### Step 5: resume で対話

修正が必要なエージェントのみに Task ツールの resume パラメータでフィードバックを送る。

#### resume分岐テーブル

| Opus側の状態 | Sonnet側の状態 | resumeの動作 |
|-------------|--------------|-------------|
| 要修正あり | 要修正あり | 両方に並列でresume（1回のレスポンス内で2つのTask呼び出しを同時発行） |
| 要修正あり | 全OK | Opusのみresume |
| 全OK | 要修正あり | Sonnetのみresume |
| 全OK | 全OK | resumeなし → Step 6へ |

※ Haiku側エージェントは resume 対象外。観点6が非準拠の場合、メインが直接プランを修正し、次ラウンドで新しい Haiku を起動して再確認する。

#### resumeプロンプトのスコープ

各エージェントへのresumeプロンプトには**そのエージェントの担当観点に関するフィードバックのみ**を含める。他方のエージェントの観点に関するフィードバックは含めない。横断チェック（Step 4）で検出された矛盾がある場合は、該当する側のエージェントのresumeプロンプトに矛盾の内容と修正後のプラン差分を含める。

resume の prompt テンプレート:

```
あなたの前回の報告に対するメインの判断を以下に示します。

## フィードバック

### 各提案への判断
- 観点N・提案内容: 採用 / 却下
  - 理由: ...
  - （採用の場合）プランの○○セクションに反映済み

### プラン差分サマリー
- 変更1: ○○を追加
- 変更2: △△を修正

### 追加チェック依頼
- （あれば）○○の観点で再確認してほしい
- （なければ）追加チェックなし

上記を踏まえて、反論がある場合は反論を、ない場合は残りの追加チェック結果を返してください。
```

### Step 6: 収束判定

**収束（完了）条件**:
- Opus側の全担当観点（1,2,5+条件付き）がOK + 反論なし
- **かつ** Sonnet側の全担当観点（3,4）がOK + 反論なし
- **かつ** Haiku側の観点6がOK（準拠）

**全観点OK時の処理**:
- 3エージェントの全観点OK + 反論なし → **Step 7（最終警戒チェック）へ進む**（スキップ禁止）
- Step 7 を実行せずに Step 8 に進むことは禁止する

**未収束（ループ継続）条件**:
- いずれかのエージェントで要修正 or 反論あり
- → Step 4 に戻る（自動ループで継続）

**自動ループの中断条件**:

以下のいずれかに該当した場合、自動ループを中断してユーザーに報告する:

1. **争点エスカレーション**: 同一争点で同一根拠の主張が繰り返された（下記の却下カウントルール）
2. **ラウンド上限**: 5ラウンドに達した（多層防御の保険。実運用では到達しない想定）

**resume について**: resume 自体に回数上限はない。検証エージェントは「完璧です」判定まで resume で生き続ける。新たな発見（修正の波及、見落とし）や中断後の再開もすべて resume で対応する。

**同一争点のラリー上限**: メインが同一争点を2回却下しても検証エージェントが同じ根拠で再度主張する場合、その争点はユーザーにエスカレーションする（自動ループ中断）。以下はカウント対象外:
- 別の根拠に基づく反論（新たなラリーとして扱う）
- 修正の波及による新規指摘（別争点）
- 以前OKだった観点での新たな発見（別争点）

### Step 7: 最終警戒チェック（Final Vigilance Check）

> **このステップのスキップは禁止**。Step 6 で全観点OKが出た場合、必ず本ステップを実行してから Step 8 に進む。

#### 目的

検証ループとは独立した視点で、プランの実装品質を調査する。
確認対象は「ニュアンスの反映」ではなく「必要なものが作られ、不要なものが消されているか」の実態。

#### 実行者

**新規Opusサブエージェント**を起動する（既存の検証エージェントとは別）。
既存検証エージェントの resume は使わない（確証バイアスの排除）。

#### 入力

- **渡す**: プランファイル、プランのファイル操作セクションから抽出した変更対象ファイル一覧
- **渡さない**: conversation.md、検証ループの結果（先入観を排除し、品質チェックに集中させるため）

#### チェック内容

ファイル存在性、内容整合性、Git追跡状態、参照整合性の4カテゴリをチェックする。
詳細はテンプレートファイルに記載。

#### Task呼び出し方法

```
以下のテンプレートファイルを読み込み、指示に従って検証を実行してください。

テンプレート: .claude/skills/planmode-rules/subagent-prompts/final-vigilance.md

入力ファイル:
- プランファイル: {plan_file_path}
- 変更対象ファイル一覧:
  （プランのファイル操作セクションから抽出したリストをここに列挙）
```

#### やり取りフロー

1. メインOpus → 最終チェックOpus起動（プラン + 変更一覧 + 上記プロンプト）
2. 最終チェックOpus → 調査実行 → チェックリスト報告を返す
3. メインOpus → 報告を精査し、承認/却下/質問を resume で返す
4. 最終チェックOpus → 反論 or 追加調査 → 更新結果を返す（resume）
5. （必要に応じてもう1ラウンド。通常2-3ラウンドで収束）
6. メインOpus → 全4カテゴリ承認 → Step 8 へ

メインOpusは既存ループと同様の構造化フィードバック形式で返す。
同一争点で2回却下後も再主張された場合は争点エスカレーション（既存ルールを適用）。

#### エージェントID管理

以下の3つのエージェントIDを保持する:

| ID | 用途 |
|----|------|
| 検証Opus ID | 観点1,2,5+条件付きの検証エージェント |
| 検証Sonnet ID | 観点3,4の検証エージェント |
| 最終チェックOpus ID | Step 7の最終警戒チェック（現行通り） |

※ Haiku側エージェントはエージェントIDを保持しない（毎ラウンド新規起動のため）。

「完璧ですか？」トリガーでの再開時は、最終警戒チェックの状態に応じて:
- Step 7 未完了（チェック進行中） → 最終チェックOpus IDを resume
- Step 7 未着手（Step 6 のループ中断から再開の場合） → 検証Opus IDと検証Sonnet IDの未収束の方をresume（両方未収束なら並列でresume）
  ※ Haiku側（観点6）が未収束の場合は、resume ではなく新規 Haiku を起動して再確認する。
- Step 7 完了済み → 検証Opus IDにresumeで修正箇所の再検証を依頼する（Opus側が判断観点を多く担当しているため、プラン変更の影響確認にはOpusが適切）。プラン変更がSonnet側担当観点（3,4）にも影響する場合は、検証Sonnet IDにもresumeで再検証を依頼する

### Step 8: 最終報告（収束時または中断時）

自動ループの収束または中断時に、検証結果を2部構成で報告する:

（※ 該当する条件付き観点があれば `[条件付き] ラベル名 | Opus | ...` の行を追加）

#### Part 2: 最終警戒チェック結果

## チェック観点

> Opusエージェント（観点1,2,5+条件付き）、Sonnetエージェント（観点3,4）、Haikuエージェント（観点6）の3並列で検証する。メインが全報告を統合判断する。

### 1. 会話ニュアンスの反映

ユーザーの発言には、明確に言語化されたものと、行間に存在する暗黙のニュアンスがある。プランに書かれているのは「言語化された結論」だけで、**その結論に至った温度感・懸念・好み**が抜け落ちていないか。

- 会話を遡り、ユーザーの各発言がプランのどこに反映されているか対応を確認する
- 「こうしたい」だけでなく「こうはしたくない」「こうなるのは嫌だ」も含まれているか
- ユーザーが言い淀んだ部分、補足的に言い添えた部分に重要なニュアンスが潜んでいないか
- プランの可読性を保ちつつ、ニュアンスを「背景」や「ユーザーの意図」セクションに織り込めているか

### 2. 新規セッション実行可能性

このプランを渡される別セッションのAIは、**今の会話コンテキストを一切持っていない**。そのAIがプランだけ読んで、ユーザーに追加質問することなく即座に作業開始できるか。

- 「なぜこの方針なのか」が書かれていないと、実行AIが別の方針を取るリスクがある
- 暗黙の前提（プロジェクト構造、既存の慣習、ツールの使い方等）が明文化されているか
- ファイルパス、関数名、具体的な変更箇所が特定できるレベルか

### 3. 指示の具体性

曖昧な指示は実行AIの「解釈」を生み、ユーザーの意図とズレた実装になる原因。

- 「適切に処理する」「必要に応じて」「よしなに」「適宜」等の曖昧表現を探す
- 各ステップで「何を」「どのファイルの」「どう変えるか」が明記されているか
- 判断が必要な箇所には判断基準が書かれているか（基準がないと実行AIが勝手に判断する）

### 4. 影響範囲の網羅性

1つのファイルの変更は、他のファイルに波及することがある。プランが変更対象として挙げているファイルだけで本当に完結するか。

- import/参照関係、設定ファイル、ドキュメント等で同時編集が必要なファイルが漏れていないか
- 必要に応じてサブエージェントで影響範囲を調査する（対象が少数で明白な場合は不要）
- サブエージェントの調査結果はメインコンテキストが検証してから判断に使う

### 5. サブエージェント設計の妥当性

サブエージェントの用途は「並列処理」だけではない。**メインコンテキストのコンテキスト消費を抑える**ために、調査や単純作業を委任する用途もある。

- 以下の用途が適切に検討されているか:
  - **並列処理**: 独立した作業の同時実行
  - **調査委任**: 広範囲の調査を委任し、メインのコンテキスト消費を抑える
  - **単純作業委任**: 定型作業を委任し、メインは判断・統合に集中
- サブエージェントの結果をメインAIが**必ず検証・解釈してから**ユーザーに報告する構造か
- サブエージェントの報告をそのまま鵜呑みにする構造になっていないか
- 逆に、サブエージェント使用がコンテキスト消費的に逆効果なケースで無理に使っていないか

### 6. テンプレートフォーマット準拠

プランファイルがテンプレート（plan-skeleton.md）の構造パターンに沿っているか。フッター定型文（実装完了チェック・コミット）の消失を検出する。

- フロントマター、必須セクション、セクション区切り線、フッター定型文が揃っているか
- 詳細なチェック項目は `subagent-prompts/haiku-format-check.md` を参照

## 条件付き観点ルーティング

条件付き観点は `references/conditions/` ディレクトリに1観点1ファイルで管理される。各ファイルは以下の固定テンプレートに従う:

- **発動条件**: いつ起動するかの詳細条件
- **検証内容**: 何をチェックするか（各観点固有の注記も検証内容セクション内に含めてよい）
- **検証プロンプト**: 検証エージェントに渡すプロンプトテキスト

起動判定は本ファイルの「条件付き観点の起動判定」セクションのルーティングテーブルを参照。

**重要**: 条件付き観点は常にOpus側エージェントのプロンプトに含める（Sonnet側には含めない）。条件付き観点ファイル内の「検証エージェント」は「検証Opus側エージェント」を指す。

### 条件付き観点の追加手順

新しい条件付き観点を追加する場合:
1. `references/conditions/` に新しいmdファイルを作成（固定テンプレートに従う）
2. 本ファイルの「条件付き観点の起動判定」セクションのルーティングテーブルに行を追加
