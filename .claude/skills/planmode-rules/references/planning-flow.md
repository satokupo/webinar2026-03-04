# プラン作成ワークフロー（Phase 1-3）

## プランモード突入時のフロー分岐

| 状態 | 開始フェーズ |
|------|------------|
| プランモード内でブレスト開始 | Phase 1 から |
| 通常モードでブレスト済み | Phase 1 スキップ → **Phase 2 から開始** |

いずれの場合も **Phase 2（合意形成YAML）は必須**。議論が十分であるほどYAML化は容易であり、省略する理由にはならない。

## Phase 1: ブレスト・検討
- ユーザーと対話しながら要件を明確化
- この段階ではプランファイルに触らない
- 「**プラン作成フェーズに入っていいですか？**」と確認してからPhase 2へ

## Phase 2: 合意形成（YAML構造化）

> **Phase 2 はプランニングの必須プロセスである。**
> 議論の結論をYAML形式で構造化し、ユーザーに提示するステップ。
> 「もう十分議論した」「要件は明確だ」と感じたなら、なおさらYAMLへの構造化は容易なはず。
> その判断を、省略の根拠ではなく、YAML出力の動機として使うこと。

- ユーザーの要望をYAML構造化して提示
- AIの回答（採用アプローチ）もYAML構造化
- **キーは必ず日本語にする**（読みやすさのため）
- YAML提示と同時に、以下の `AskUserQuestion` でコンテキスト圧縮の要否を確認する:
  - **question**: 「プラン作成に進みます。コンテキスト圧縮の方法を選択してください」
  - **header**: 「プラン準備」
  - **multiSelect**: false
  - **options**:
    1. **label**: 「コンテキストを圧縮してからプラン作成 (Recommended)」
       **description**: 「context-handoverで引き継ぎファイルを作成し、コンパクション後にプラン作成に入ります」
    2. **label**: 「直接プラン作成」
       **description**: 「コンテキスト圧縮せずにプラン作成に入ります（短い相談で圧縮が不要な場合）」
  - 選択肢1 → Phase 3 の Step 0（context-handover）を実行してからプラン作成
  - 選択肢2 → Phase 3 の Step 0 をスキップしてプランファイル作成手順に直接進む
- **YAML提示後にテキストでの「OKですか？」確認は不要**。AskUserQuestionの選択がYAML承認を兼ねる
- AskUserQuestionの回答を受けたらPhase 3へ進む。YAMLに修正がある場合、ユーザーは「その他」で修正指示を伝える
- 「プランに書く内容の詳細」を別途提示する必要はない（合意形成YAMLで十分）

### 省略できる唯一の条件

Phase 2 を省略できるのは、ユーザーが以下のような発話で**自らの言葉で明示的に省略を指示した場合だけ**である。

該当する発話の例:
- 「YAMLはスキップして」
- 「合意形成は締結済みとして進めて」
- 「Phase 2は飛ばしていい」

**AI自身がどのような根拠を思いついても、省略の判断をしてはならない。**
省略の権限はユーザーにのみある。AIにはない。

> **典型的な自己合理化パターン（参考）**:
> 以下はAIが生成しがちな省略理由だが、いずれも無効である:
> 「ブレストで十分議論した」「要件が明確」「会話の流れで合意が取れている」
> 「ユーザーが急いでいる」「簡単な変更なのでYAMLにするまでもない」
> これらに限らず、AI発の省略理由はすべて無効。

## Phase 3: プランファイル作成

### Step 0: コンテキスト引き継ぎ（context-handover）

> **コンパクションの目的**: プラン作成の品質を向上させるための品質戦略。
> Phase 1-2 のブレスト・合意形成でコンテキストに蓄積された低密度な情報を、
> 高密度な引き継ぎファイルに集約し、S/N比を改善する。
> これにより、Phase 3 のプラン作成時に以下の効果が期待できる:
> - lost-in-the-middle 回避（引き継ぎファイルが additionalContext として先頭配置される）
> - コードベース調査のコンテキスト余裕確保（Read/Grep/Glob 多用への備え）
> - 引き継ぎファイルの「次にやるべきこと」によるプラン作成の方向づけ

プラン作成開始前に `context-handover` スキルを Skill ツールで発動する。
スキルが引き継ぎファイルを生成し、/compact を自動送信する。

コンパクション完了後、planning-flow.md を再度 Read してプランファイル作成手順に進む。

※ 本ステップは planmode-rules スキルの再発動なしで継続可能。
  SessionStart フックが引き継ぎファイルを additionalContext に注入するため、
  メインコンテキストは状態を自動復元できる。

※ 以下の場合は本ステップをスキップする:
  - Phase 2 末尾の AskUserQuestion で「直接プラン作成」が選択された場合
  - summon-experts 経由で「直接プラン作成」が選択された場合
  上記以外（Phase 2 がスキップされ、明示的な選択がない場合を含む）は本ステップを実行する。

> **前提条件**: Phase 2 で合意形成YAMLをユーザーに提示し、AskUserQuestionで次ステップが選択されていること。
> この前提を満たさずに Phase 3 に進むことはできない。

### プランファイルの書き出し手順（必須）

プランファイルの作成には、Claude Code の正規手順を必ず使用すること。

**メカニズム**: `EnterPlanMode` を呼び出すと、システムがプランモード状態を管理し、プランファイルのパス情報をセッションに登録する。`ExitPlanMode` はこの登録情報を参照してプランを読み込む。

**パス検証（必須）**: プラン記述を開始する前に、システムが提示したパスが以下の条件を満たすことを確認する:
- パスが `<プロジェクトルート>/.claude/plans/` 配下であること
- パスにスキルディレクトリや src/ 等のサブディレクトリが含まれていないこと
- CLAUDE.md の「プロジェクトルート」セクションに記載された絶対パスと照合し、提示されたパスがその絶対パスの `.claude/plans/` 配下であることを確認する（例: プロジェクトルートが `<プロジェクトルート>` なら、パスは `<プロジェクトルート>/.claude/plans/` で始まること）

パスが異常な場合（例: `skills/xxx/.claude/plans/` のようなパスが提示された場合）は、**プラン記述を中断してユーザーに報告する**。自力でパスを修正して Write で書き込んではならない。

**手順**:
1. `EnterPlanMode` でプランモードに入る → システムメッセージに「Plan File Info」としてプランファイルのパスが表示される。**このパスにのみプランを記述すること。**
2. パス検証を実行（上記ルール通り）
3. テンプレートをプランファイルにコピー:
   ```bash
   cp .claude/skills/planmode-rules/templates/plan-skeleton.md <プランファイルパス>
   ```
4. Edit で `{{planning-session}}` → セッションIDに置換（`current-session-id` スキルで取得）
5. Edit で `{{created}}` → 日付に置換
6. Edit で `(タイトルをここに記入)` → 実際のタイトルに置換
7. Edit で各セクションのプレースホルダーを実際の内容に置換
8. 不要な推奨/任意セクション（「不要なら見出しごと削除」の注記があるもの）を見出し・区切り線ごと削除
9. `ExitPlanMode`

**禁止**:
- パスを自力で推定・構築して Write ツール等でプランファイルを作成すること。EnterPlanMode を経由してシステムが提示したパスのみを使用する
- Write で全文を一括書き出すこと（テンプレートの `cp` → Edit による差分更新を必須とする）

### プランの固定構造

プランファイルは以下のヘッダーとフッターを**省略不可の定型要素**として持つ。変更規模に関わらず、すべてのプランはこの固定構造を含むこと。

#### ヘッダー: フロントマター

テンプレート（`templates/plan-skeleton.md`）のフロントマターを参照。セッションIDは `current-session-id` スキルで取得する。

#### フッター: 実装完了チェック＋コミットセクション

テンプレート（`templates/plan-skeleton.md`）のフッターを参照。定型文面の Single Source of Truth はテンプレートファイルである。

#### セクション構成

プランの本文は以下のセクション構成を基本とする。必須セクションは規模に関わらず含めること。推奨・任意セクションはプランの内容に応じて調整してよい。

**必須セクション**（この順序で配置）:
1. `## 📝 背景` — 問題意識、経緯、ユーザーの意図・好み、検討経緯を含む
2. `## 📄 ファイル操作` — 操作対象ファイルの一覧（後述のフォーマットに従う）
3. `## 🔧 変更内容` — 詳細な実装指示
4. `## ✅ 実装完了チェック` — 実装完了の検証手順（上記フッターの定型文）
5. `## 🚀 コミット` — コミット手順（上記フッターの定型文）

**推奨セクション**（基本的に含めるが、内容が自明な場合は省略可）:
- `## ✅ 検証方法` — 変更内容の後、コミットの前に配置
- `## 📌 スコープ外の課題` — 検証方法の後、コミットの前に配置。該当なしの場合は「なし」と明記

**任意セクション**（プランの内容に応じて適切な位置に追加）:
- 修正不要の確認、並列処理指示、設計判断の詳細など
- 背景に関連するものは背景セクション内にサブセクションとして追加
- 実装に関連するものは変更内容セクション内にサブセクションとして追加

#### セクション区切り線

`##` レベルの主要セクション見出しの**上側**に、ASCII イコール40文字の区切り線を配置する。これは視覚的な補助であり、セマンティックな構造は `##` 見出しが担う。

**フォーマット**:
- 前セクションの末尾から**空行2行**
- 区切り線: `========================================`（イコール40文字）
- 区切り線の後に**空行1行**
- `##` 見出し（エモジ付き）

**適用範囲**: `##` レベルの見出し間のみ。`###` 以下のサブセクション間には適用しない。

※ フロントマター直後の最初の `##` セクション（背景）の上にも区切り線を入れる。

#### ファイル操作セクションのフォーマット

操作種別ごとにカテゴリ分けし、各ファイルをフルパス + 一行説明で記載する。

**カテゴリ**: 新規作成 / 編集 / 削除（該当がないカテゴリは省略する）

**フォーマット例**:

    ## 📄 ファイル操作

    **新規作成**:
    - `claude/skills/example/references/guide.md` — 操作ガイドを新規作成

    **編集**:
    - `claude/skills/example/SKILL.md` — usage フィールドの説明を追加
    - `claude/skills/example/scripts/main.py` — エラーハンドリングを改善

    **削除**:
    - `claude/skills/example/scripts/old_helper.py` — helper.py に統合のため不要

**ルール**:
- ファイルパスはバッククォートで囲む
- 説明は ` — `（スペース+emダッシュ+スペース）で区切る
- 説明は動詞始まりを推奨（「追加する」「改善する」「不要のため削除」等）
- パスはプロジェクトルートからの相対パス

### issueファイルの取り扱い制約

プランに `.claude/context/issues/` 配下のファイルに関する操作を含める場合、以下のルールに従うこと:

- **issueファイルの削除は禁止**。解決済みのissueは削除ではなくアーカイブに移動する
- **アーカイブ先**: `.claude/context/issues/archive/`
- アーカイブディレクトリが存在しない場合は、ユーザーに「どこにアーカイブしますか？」と確認する（勝手にディレクトリを作成しない）
- issue解決を含むプランには、解決後のissueファイルの扱い（アーカイブ移動 → コミット）を必ず明記する

### リポジトリ外への書き込み制約

プランの検証手順やスクリプト実行等で、**現在のリポジトリ外かつ dotfiles 外**の場所（`/tmp/` 等）にファイルやディレクトリを作成・変更する必要がある場合:

- **プラン作成時**: プランにそのような操作を含める前に、**必ず AskUserQuestion** でユーザーに明示的に確認する。テキスト中に埋め込んで聞くだけでは見落とされるため、AskUserQuestion による確認を必須とする
- **許可された場合のみ**プランに記載し、該当箇所に「ユーザー確認済み」と注記する
- dotfiles（`~/dotfiles/` や `~/.claude/`）内は他リポジトリからも参照されるため、この制約の対象外

### 本文の記述ガイドライン

**前提**: プランは別の新しいセッションで実行される可能性が高い。実行時のAIはプランニング時の会話コンテキストを持たないため、**プラン単体で意図が伝わる詳細さ**が必要。

**含めるべき内容と対応セクション**:

`## 📝 背景` セクションに含める:
1. **背景・問題意識** - ユーザーが何を解決したかったか、現状の課題
2. **やりとりの要約** - 重要な発言・指摘、ユーザーが強調したポイント、細かいニュアンス
3. **検討の経緯** - 疑問→検討→結論の道筋、選択肢の比較、採用/却下理由
4. **ユーザーの意図・好み** - 暗黙の期待、「これは避けてほしい」等の価値観

`## 🔧 変更内容` セクションに含める:
5. **実装方針** - 具体的な変更内容、ファイルパス、手順

**心得**:
- 「何を」だけでなく「なぜ」を必ず書く
- 新セッションのAIが「なぜこの方針なのか」を理解できるレベルに
- 迷ったら詳しく書きすぎるくらいでよい

### 記述例

悪い例（結果のみ）:

    ## 変更内容
    - configにオプションAを追加する

良い例（経緯を含む）:

    ## 背景
    ユーザーは設定の柔軟性を求めていた。「毎回手動で変えるのが面倒」という発言あり。

    ## 検討経緯
    - 案1: 環境変数で切り替え → ユーザー「コマンドラインで完結させたい」
    - 案2: configファイルにオプション追加 → 採用（理由: 永続化でき、ユーザーの好みに合致）

    ## 変更内容
    - configにオプションAを追加する

### 大規模プランの場合

以下の条件に該当する場合は、**プランに並列処理の指示を含める**:
- 変更対象ファイルが3つ以上
- 複数の独立した作業ステップがある
- 作業間でコンフリクトしない内容

プランには「メインコンテキストをオーケストレーター（司令塔）とし、サブエージェントに並列で作業を割り振る」旨を記載する。実行セッションのAIがこれを読んで並列処理を実行する。

### サブエージェントを使用するプランの場合

プラン実行時にサブエージェント（Taskツール）を使用する可能性がある場合、**以下の申し送り事項をプランファイルに含める**：

テンプレート内の `### サブエージェント使用時の注意` セクションに定型文が含まれている。使用しない場合は小節ごと削除する。
