# Codexレビューフロー（Phase 5 オプション）

## トリガー
- ユーザーが「Codexにレビューして」「Codexレビューお願い」等と明示的に発言した場合のみ
- Phase 4完了（「完璧です」判定済み）が前提条件
- **自動遷移・自動推奨は一切禁止**

## 前提条件チェック
- Codex CLIがインストールされているか（`which codex`）
- Phase 4が完了しているか

## Step 1: Codexレビュー実行

codex_reviewer.py の review サブコマンドを実行:

```bash
python3 .claude/skills/planmode-rules/codex_reviewer.py review \
  --plan <プランファイルパス> --cwd "$(pwd)" --session-id <セッションID>
```

※ Codex CLI は `--search` フラグ（live モード）で実行される。レビュー中に最新の技術情報や脆弱性データが必要な場合、Codex が自発的に Web 検索を行う。

- タイムアウト: Bashツールの timeout を 600000（10分）に設定
- 出力: スレッドID（`codex_thread_id`）+ レビュー結果ファイルパス（.claude/temp/{session_id}_codex-review.md）

## Step 2: OPASトリアージと対話

### 2-1. レビュー結果ファイルを読み、各指摘を評価

- **採用**: 指摘が妥当で改善につながる → プランに反映
- **却下**: オーバースペック、ユーザー意図と乖離 → 理由を記録
- **確認**: 不明点がある → Step 3 で追加質問

※ 「確認」判定の指摘がある場合は、先に Step 3 の追加質問を完了し、
  全指摘の採用/却下が確定してから Step 2-2 に進む

### 2-2. 判断結果を Codex に返し、反論を求める

codex_reviewer.py の followup サブコマンド（resume）で以下のプロンプトを送信:

```
あなたのレビューに対する判断結果を以下に示します。

## 各指摘への判断
- 観点N: 採用 / 却下
  - 理由: ...
  - （採用の場合）プランへの反映内容

## 反論要求
却下した指摘について、別の根拠に基づく反論があれば述べてください。
同じ理由の繰り返しは不要です。
反論がなければ「反論なし」とだけ回答してください。
新規の観点がある場合は「追加指摘」として別途記載してください。
```

### 2-3. Codex の反論を受け取る

- **反論なし** → Step 4 へ
- **反論あり** → 再評価し、最終判断を確定
- **追加指摘** → 一方通行で採用/却下を判断（追加ラウンドトリップなし）

## Step 3: 追加質問（必要な場合のみ）

codex_reviewer.py の followup サブコマンドを実行:

```bash
python3 .claude/skills/planmode-rules/codex_reviewer.py followup \
  --codex-session <CodexスレッドID> --question "質問テキスト" --cwd "$(pwd)" --session-id <セッションID>
```

- 追加質問は最大2回まで
- 2回を超える場合はユーザーに状況を報告し判断を委ねる

### followup 回数ルール

- 反論要求（Step 2-2）: 1回
- 追加質問（Step 3）: 最大2回
- 合計: followup 最大3回
- スクリプト側では制限しない。Claude Code 側で管理する

## Step 4: プラン修正とユーザー報告

修正規模で3段階に分岐:

- **軽微**（ユーザー確認不要な範囲: 文言修正、注意事項1〜2行の追加）:
  OPASがプラン修正して完了。修正箇所をユーザーに報告
- **中程度**（ユーザー確認が必要な範囲: 新しいセクションの追加、手順の変更等）:
  OPASがプラン修正し、修正内容をユーザーに報告して確認を取る
- **大幅**（アーキテクチャ変更、方針の根本的見直し）:
  ユーザーに報告し、確認を待つ。
  ※ Phase 4 エージェントIDの引き継ぎ機構は未実装。
    現時点では、大幅な修正が必要な場合はユーザーに状況を報告し、
    Phase 4 の再実行（「検証準備して検証して」）を提案する運用とする。

判断基準: プランの実装方針セクションの書き直しが必要かどうか。
注意事項追記なら方針は変わっていない（軽微〜中程度）。方針そのものの変更なら大幅。

### 報告テンプレート

修正規模に関わらず、以下のテンプレートで報告する:

```markdown
### Phase 5 Codex レビュー完了報告

**結論**: 指摘N件中、採用X件・却下Y件
- 変更点1: ...
- 変更点2: ...

#### 1. Codex からの指摘
- 観点1: 指摘内容の要約
- 観点2: 指摘内容の要約

#### 2. Claude Code の判断
- 観点1: 採用 — 理由: ...
- 観点2: 却下 — 理由: ...

#### 3. Codex へのフォローアップ結果
- Codex は「反論なし / 反論あり（内容）」と回答
- （反論があった場合）再評価結果: ...

#### 4. プランへの最終反映
- 採用した指摘に基づく変更内容

Codex スレッドID: `{thread_id}`
API使用量: 入力 {input} / 出力 {output} トークン（概算 ${cost}）

📁 **レビュー結果ファイル**: `{review_file_path}`
⚠️ このファイルは一時ファイルです。チャットログ保存時（save-chatlog）に自動削除されます。保存が必要な場合はコピーして別場所に移動してください。
```

### API使用量の報告ルール

review および各 followup の JSON 出力に含まれる `usage` フィールドからトークン数を取得する。
複数回の呼び出し（review + followup 最大3回）がある場合は、全呼び出しの usage を累計する。

概算式:
`cost ≈ (input_tokens / 1,000,000) × $1.75 + (output_tokens / 1,000,000) × $14.00`
※ gpt-5.2-codex 基準の参考値（2026年2月時点）。モデル変更時は単価を更新すること。

- usage が取得できた場合: 報告テンプレートに「API使用量」行を含める
- usage が取得できなかった場合（`usage` が `null`）: 「API使用量」行を省略する（エラー表示はしない）

変更点リストは **Codex の指摘単位（観点単位）** で記載する。1つの指摘に複数の変更が伴う場合はまとめて1項目とする。

報告後のユーザー確認要否は修正規模で分岐:
- 軽微: 報告して完了
- 中程度: 報告後にユーザー確認を待つ
- 大幅: 報告後にユーザー確認を待つ

## 「レビュー完了」サウンド

Step 4 の報告テンプレート（「Phase 5 Codex レビュー完了報告」）の出力が完了した時点で、以下の完了動作を実行する。

**完了時の動作**:
1. 「🛡️🛡️ Codexによるレビューが完了しました 🛡️🛡️」と回答する
2. `.claude/skills/planmode-rules/play_sound.py` を実行して達成サウンドを再生する

※ 手順1のテキスト出力が完了してから手順2を実行すること（並列実行禁止）
